# 试剑石 修正计划 — 从"找注射bug"到"诊断真实问题"

> 借鉴：SWE-bench (OpenAI), DSPy (Stanford), Delta Debugging (Zeller), Anthropic Eval Philosophy
> 核心洞察：当前方法在测量"AI能否找到你藏的东西"，不是在测量"AI能否debug"

---

## 一、诚实诊断：当前设计 4 个致命缺陷

### 缺陷 1：构造效度问题 (Construct Validity)

**现象：** B03(2/8) + B04(2/8) — agent 发现了真实 bug 但被判错。

**根因：** GT (ground truth) 是"注入 bug 的指定位置"。agent 找到的 bug 是真的但不匹配 GT → 低分。高分 agent 被系统性惩罚"太聪明"。

**类比：** 考试问"第3页第5行的错别字是什么"，学生说"第2页有个逻辑错误"→ 判零分。学生是对的，但考试设计有问题。

**SWE-bench 怎么做的：** 用真实 GitHub issue + 真实修复 PR。评判标准是"AI 的修复是否通过相同测试"，不是"是否找到指定位置"。

### 缺陷 2：合同链是线性的，debugging 是螺旋的

**现象：** [分类]→[证据]→[追踪]→[分析]→[修复]→[验证]→[记录]。v2.6 试图重排序为 [证据]→[分类]→[追踪] 但退化了。

**根因：** 专家 debugger 不按线性步骤工作。她们在证据、追踪、分析之间来回跳跃，不断修正假设。线性的合同链迫使 agent 过早锁定分类。

**认知科学依据：** Katz & Anderson (1987) 证明 debugging = hypothesis testing。Zeller (2002) 的 Delta Debugging 也是迭代缩小差异，不是一步分类。

### 缺陷 3：评分系统有确认偏误

**现象：** Judge 和 agent 用同一模型 (deepseek-v4)。同模评自己 → 系统性偏误。

**根因：** 7 维平分(每维 1-2 分)假设所有维度同等重要。实际上"根因正确"比"分类正确"重要得多。一个 agent 分类错了但根因对了 = 实际上成功。分类对了但根因错了 = 实际上失败。

**Anthropic Eval Philosophy：** 多评估者、多指标、人类基线、对抗测试。我们一个都没有。

### 缺陷 4：成长引擎过度设计，执行不足

**现象：** 5 环引擎只有 R1 真正工作。R2/R3/R4/R5 因数据格式问题从未触发。

**根因：** 过度设计。在只有 8 次 run 的数据量上建 5 环引擎，就像用 8 个样本训练神经网络。

**Google 内部研究：** 简单启发式 trigger + 人工审查 > 复杂自动系统，尤其在数据稀疏时。

## 二、理论地基 — 借鉴谁、借鉴什么

### 借鉴 1：SWE-bench — 真实 bug，不是注入 bug

| SWE-bench | 我们的修正 |
|-----------|----------|
| 真实 GitHub issue 作为 bug 描述 | 从 git history fix commits 提取真实 bug |
| 真实 PR diff 作为 GT | 实际 fix commit 的 diff 作为 GT |
| 测试通过率作为评分 | 修复有效性（测试 PASS + 无回归）+ 根因正确性 |
| 人工验证每个 case | 凡哥 5min/case 审 mine 产出 |

### 借鉴 2：DSPy — 优化 program，不是优化 prompt

| DSPy | 我们的修正 |
|------|----------|
| Prompt = Program，需要编译优化 | Skill = 可执行合同，需要数据驱动迭代 |
| 自动生成候选优化 → 验证 → 保留显著的 | GEPA teacher 生成候选 F-rule → T2 验证 → 保留有效果的 |
| Hold-out 验证防止过拟合 | Hold-out 10 bugs 每轮验证 gap |

### 借鉴 3：Delta Debugging (Zeller) — 螺旋缩小，不是树形分类

| Delta Debugging | 我们的修正 |
|----------------|----------|
| 系统性缩小"通过 vs 失败"的差异 | 合同链改为螺旋：假设 → 验证 → 修正 → 再假设 |
| 不预分类，渐进逼近根因 | T0-T7 从入口关变为出口标签 |
| 每次测试减少一半差异 | 每次假设验证排除一类根因 |

### 借鉴 4：Anthropic Eval Philosophy — 多维度，不单指标

| Anthropic | 我们的修正 |
|-----------|----------|
| Multi-evaluator (多个独立评估者) | 双模型 judge (deepseek + sonnet)，分数差 >3 → 人工 review |
| Human baseline (人类参照) | Phase 0: 凡哥手动做 5 bugs，得锚点 |
| Red-teaming (对抗测试) | Trap bugs 保留，新增 "discovered real bug" bonus |

## 三、修正后的架构

### 3.1 螺旋合同链（替代线性链）

```
[Bug描述] → [初始假设H1] → [快速验证] → [修正假设H2] → [深度验证] → [根因确认] → [修复]
    ↑         ↑                                                              |
    |         └── H1被推翻：记录"为什么H1错了" → 生成H2                        |
    └────────── H2也被推翻：回到修正，最多3轮 ─────────────────────────────────┘
```

**关键规则：**
1. 每个假设必须可证伪："如果我是对的，X应该为真。现在检查X。"
2. 假设被推翻时强制记录："H1是___，被___证据推翻。H2改为___。"
3. 3轮假设还没确认 → 触发 Red Line 4（质疑架构，可能根因类别完全错误）
4. **T0-T7 不再作为入口分类，而是出口标签** — 在根因确认后贴标签，用于统计分析
5. F1-F6 致命误判表保留，但角色变为"假设被推翻时的替代假设来源"

### 3.2 新评分系统（替代 7 维平分）

| 维度 | 权重 | 0分 | 满分标准 |
|------|------|-----|---------|
| **根因正确性** | 0-4 | 完全错误 | 精确命中真实根因 |
| **修复有效性** | 0-3 | 无效/引入新bug | 完全修复，无回归 |
| **过程质量** | 0-2 | 跳步/无假设记录 | 假设链完整，推翻记录清晰 |
| **真实发现 bonus** | +1 | — | 发现并正确报告了非GT的真实bug |

**总分：0-10 per bug。** 根因权重最高 (40%)，因为这是 debugging 的核心。

**Judge 改进：**
- 同一 run 用 deepseek + sonnet 各评一次，取均值
- 两模型分数差 >3 → 标记 `[JUDGE_DISPUTE]`，需凡哥人工裁决
- 成本：+$0.25/run（一次额外的 judge 调用）

### 3.3 真实 Bug 迁移路径

```
Phase A ($0):     注入 49 bugs (保留) + 真实 10 bugs (新增)
Phase B ($0/次):  每次 mine 追加 2-5 个真实 bug
Phase C (达标):   真实 ≥30 → 退役大部分注入 bug
```

真实 bug 来源：98 个 fix commits → LLM 提取 → 凡哥 5min 审 → 入 R01-Rxx 系列。

### 3.4 简化成长引擎（5 环 → 3 环）

| 保留 | 砍掉 | 理由 |
|------|------|------|
| R1 挖矿 (git mine) | — | 唯一工作的环，持续产出 |
| R4 过拟合 (hold-out gap) | R2 退役 (3x完美) | 数据量不足，阈值不可靠 |
| R5 退化 (score drop) | R3 盲区 (c4=0) | per-bug 数据太少 |

新增触发：**R0 假设失败模式** — 同类型假设被推翻 3+ 次 → 候选新 F-rule。

## 四、执行计划（3 Phase, $3 预算）

### Phase 0: 人类基线（$0, 必须第一步）

**为什么先做：** 不知道 95.4% 对人类是什么水平。可能是 80%（AI 已超人类），也可能是 99%（AI 还有路）。

**操作：**
1. 选 5 bugs：T0(简单crash) + T3(数据错误) + T4(配置问题) + T7(NOT_A_BUG) + 1 trap
2. 凡哥手动做：不限时，不限工具。记录：分类 / 根因 / conf / 耗时
3. AI 用 3.2 新评分系统打分
4. 得锚点分数 → 所有后续 AI 分数以此为参照

**预期：** 人类 7-9/10 per bug (50-90% range)。如果人类 9+ → 天花板高，AI 还有路。如果人类 6-7 → AI 已超人类，重点转为 reliability。

### Phase 1: 基础建设（$0）

1. **改 bughunt_t2.js** → 输出格式改为新评分系统 (0-10 per bug)
2. **改 bughunt_t2.js** → agent 用螺旋合同链 v3.0-alpha
3. **写 bughunt_real.js** → 10 个真实 bug (R01-R10) 的专用脚本
4. **提取 10 个真实 bug** → 从 git history → LLM 脱敏 → 凡哥审
5. **改 judge** → 双模型 (deepseek + sonnet)，均值，dispute flag

### Phase 2: 螺旋模型验证（$2.00）

1. **v3.0-alpha × 5 注入 bugs × 1 run ($0)** → dry-run，确认螺旋链可执行
2. **v3.0-alpha × 10 真实 bugs × 1 run ($0.50)** → 真实测试
3. **v3.0-alpha × 10 注入 bugs × 3 runs ($1.50)** → 方差测量
4. **v2.5.1 × 同样 10 bugs × 1 run ($0)** → head-to-head 对比
5. **判定：** v3.0 ≥ v2.5.1 → 替换。v3.0 < v2.5.1 → 保留 v2.5.1，螺旋模型回炉。

### Phase 3: 闭环（$1.00）

1. Hold-out 测试（螺旋模型 + 真实 bugs）
2. Cross-model judge 验证
3. 第一个自动 GEPA → F-rule → 验证的完整闭环

## 五、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 螺旋模型 agent 步数增加 → 成本上升 | 高 | 中 | 设 max 3 轮假设，超限 fallback 线性的 v2.5.1 |
| 真实 bug GT 不精确（commit message 模糊） | 中 | 高 | 凡哥审 + 只选 message 清晰的 commit |
| 新评分系统主观性 > 旧系统 | 中 | 中 | 双 judge + dispute flag + 人类校准 |
| v3.0 不如 v2.5.1（类似 v2.6 退化） | 中 | 高 | Dry-run 5 bugs first，不比 v2.5.1 差才扩到全量 |
| 凡哥没时间做 human baseline | 高 | 低 | 可以先跑 AI 部分，human baseline 延后但不跳过 |
| 新评分系统与旧 leaderboard 不兼容 | 低 | 低 | 保留旧评分并列运行，逐步迁移 |

## 六、成功标准

| 不是 | 而是 |
|------|------|
| 一次高分 (95%+) | 分数随真实 bug 增加保持稳定 |
| 注入 bug 全对 | 真实 bug 修复率高 + hold-out gap <5pp |
| 自动进化跑起来 | 每个自动改进经过验证（不比 baseline 差） |
| 复杂的多环引擎 | 简单的 trigger + 一致的人工审查 |
| AI 自己评自己 | 双模型 judge 一致 + 人类校准 |

---

## 七、v2.5.1 的命运

v2.5.1 保留作为 **production fallback**。不改它，不删它。

- 如果 v3.0 螺旋模型成功 → v2.5.1 退役
- 如果 v3.0 失败 → v2.5.1 仍是 active skill
- 永远不删除已证明可用的版本

## 八、诚实自问（每个 Phase 完后）

1. "我们现在测量的是 debugging 能力，还是找藏东西的能力？"
2. "这个改进有没有在真实 bug 上验证过？"
3. "人类会不会犯同样的错？如果会，这不是 AI 的问题。"
4. "我们在让 benchmark 更难，还是在让 skill 更好？"
5. "有没有凡哥看到会说'这不对'的地方？"
