# 试剑石 成长闭环 — 简单、可验证、自进化

> 借鉴: DSPy GEPA (genetic evolution), SWE-bench (continuous hardening), Expert-Executor (review gate)
> 核心: 不是 5 环引擎。是一个循环: 采集→分析→生成→验证→决定。

---

## 为什么旧 5 环引擎死了

| 旧设计 | 为什么失败 |
|--------|----------|
| R1 挖矿 | ✅ 唯一工作的环 |
| R2 退役 | 8 次 run 不够退役任何 bug |
| R3 盲区 | per-bug 数据不存在 |
| R4 过拟合 | 阈值拍脑袋, 从未触发 |
| R5 退化 | 退化到 79% 都不触发(基线设太高) |

**根因: 数据太少, 环太多。过度设计。**

## 新成长循环: GEPA-GATE

```
                    ┌─────────────┐
                    │  生产 Skill  │ ←──────────────────┐
                    └──────┬──────┘                      │
                           │                             │
                    ┌──────▼──────┐                      │
                    │  T2 诚实跑   │                      │
                    │ (per-bug)   │                      │
                    └──────┬──────┘                      │
                           │                             │
                    ┌──────▼──────┐                ┌─────┴─────┐
                    │  发现失败    │                │  MERGE    │
                    │  ≤5/8 ×2次  │                │  合入生产  │
                    └──────┬──────┘                └─────▲─────┘
                           │                             │
                    ┌──────▼──────┐                      │
                    │  GEPA分析   │                ┌─────┴─────┐
                    │  Teacher    │                │  PASS     │
                    │  生成候选   │                │  ≥baseline│
                    └──────┬──────┘                └─────▲─────┘
                           │                             │
                    ┌──────▼──────┐                ┌─────┴─────┐
                    │  单独验证   │ ──FAIL──→      │  DISCARD  │
                    │  T2 ×3 runs │                │  丢弃     │
                    │  只测该bug  │                └───────────┘
                    └─────────────┘
```

### 规则

1. **触发**: 同一 bug 连续 2 次 T2 总分≤5/8 → 标记为"生长候选"
2. **分析**: GEPA teacher 读取两次失败的 agent report + judge 评语 → 生成 1-3 个候选改进
3. **验证**: 每个候选 → 应用到 skill 副本 → T2 跑 3 次（只跑该 bug type 的 bugs, $0.15）
4. **决定**: 
   - 候选 mean ≥ baseline + 1pp → MERGE
   - 候选 mean < baseline → DISCARD
   - 多个候选都通过 → 合并验证一次
5. **记录**: 每次尝试写入 growth.log: 日期/候选/分数/决定

## 候选改进类型

| Type | 内容 | 示例 |
|------|------|------|
| **F-rule** | 新致命误判 | F7: "Agent 找真 bug 但 GT 指向另一方向→检查是否真 bug" |
| **Worked Example** | 新增或增强例子 | T2 B03 类型的新 worked example |
| **Gotcha** | 新正向检查 | "T2 调查前先确认 bug 描述中的技术栈标签" |
| **Decision** | 决策流调整 | 新增 Q10: "发现真 bug 但与 GT 方向不同→标记 bonus" |

## 预算

| 步骤 | 成本 | 频率 |
|------|------|------|
| 触发 (T2 诚实跑) | $0.50/次 | 每次 skill 改后 |
| 分析 (GEPA teacher) | $0.05 | 每个候选 bug |
| 验证 (T2 ×3 on subset) | $0.15 | 每个候选 |
| 合并验证 | $0.50 | 多候选合并时 |

**月预算: ~$5 (10 次 T2 + 3-5 候选验证)**

## 与旧引擎对比

| | 旧 5 环 | 新 GEPA-GATE |
|---|---------|-------------|
| 环数 | 5 | 1 |
| 自动触发 | 3/5 从未工作 | 1/1 数据驱动 |
| 验证 | 无 | 每个候选 T2 3-run |
| 回滚 | 手动 | 自动 (DISCARD) |
| 可解释 | 差 | growth.log 完整记录 |
| 成本 | $0 (不跑) | $0.65/候选 |

## 成长度量

```
growth.log 追踪:
- 总候选数 / 通过数 / 合并数
- skill 分数趋势 (每次 T2 后的 mean±SD)
- 最近一次改进带来的 Δpp
- 未解决 bug 列表

每月 review:
- skill lift vs bare (T3, $5)
- hold-out gap
- 新 bug 加入数 (mine)
```
