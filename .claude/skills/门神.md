# Quality Gate v2.2 — 通用质量门禁

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 可配置质量门禁链→加权分数→APPROVED/WARNED/BLOCKED。集成风险分级+中止条件。
**Iron Law：** NO DEPLOY WITH QUALITY SCORE < 80. 无例外——热修复也不行。
**红线：** 任一道FAIL→立即停不等。绝不修改分数阈值。绝不跳过安全检查(热修复也不行)。质量分<80→绝不部署。
**触发：** quality check / ready to ship / 质量门禁 / pre-deploy / release check
**边界：** 质量门禁→门神。部署→布阵。安全审计→铁壁。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | 编译过=能上线 | 编译≠功能正确 | 回归测试独立检查 |
| 2 | nucle扫nginx=应用层安全 | 应用层绕过nginx | 直连后端端口再扫 |
| 3 | 分数边界"差一分没事" | 阈值是防线 | <80绝不部署 |
| 4 | "热修复跳过安全扫描" | 热修复最危险 | 热修复也要全量扫描 |
| 5 | 两个agent一致=正确 | 两个都漏同一问题 | 分歧时加第三个 |
| 6 | 防御代码存在=防御有效 | 速率限制被注释DISABLED FOR TESTING | 不仅检查代码存在，还要验证实际生效。curl测试>静态分析 |
| 7 | nginx有限流=所有路径安全 | /campus/前缀绕过/api/login限流 | 每个location/路径独立验证速率限制 |
| 8 | 阈值4.0够了 | template输出也能得3-4分 | 验证阈值对最差case有效——构造空报告→跑gate→确认FAIL |

## 可配置检查链（+风险分级——来自deploy精华）

| 检查 | 权重 | 什么 |
|------|------|------|
| 语法/编译 | 20 | 所有语言lint/build |
| 功能测试 | 20 | 项目回归脚本 |
| 密钥扫描 | 10 | gitleaks/TIER1 grep |
| SAST | 10 | semgrep/手动GREP |
| 漏洞扫描 | 5 | nuclei/替代 |
| 多Agent共识 | 10 | 2+agent交叉验证 |
| 修复验证 | 5 | 反模式扫描 |
| 回滚就绪 | 5 | 备份<1h+回滚步骤 |

**风险分级影响检查严格度：**
```
SEV1(热修复): 全部检查+0容错(任一项FAIL=BLOCKED)
SEV2(常规):   全部检查+≤1项WARNED可过
SEV3(金丝雀): 核心检查+宽松阈值
```

## Gate Finding 严重度分级

| 级别 | 定义 | 示例 |
|------|------|------|
| 🔴 CRITICAL | 门禁漏过会导致生产故障 | 阈值4.0放过template输出、gitleaks只扫staged |
| 🟠 HIGH | 检查配置有盲区 | 覆盖率阈值可静默修改、SAST只扫部分文件 |
| 🟡 MEDIUM | 单点薄弱 | 单一检查无fallback、假阳性未筛选 |
| 🟢 LOW | 优化建议 | 检查顺序可调整、权重微调 |

## 阈值完整性审计（Q03 pattern — 每次门禁必查）

```
1. 列出所有阈值文件: codecov.yml, bughunt_ci.py GATE_THRESHOLDS, lefthook.yml
2. 检查每个阈值: 变更是否需要审批？CI是否检测阈值本身被修改？
3. 验证: 改一个阈值→提交→CI是否能检测并告警？
4. 修复: 阈值文件hash锁定 + CI step: "if threshold file changed → require approval"
```

## Hook 覆盖审计（Q02 pattern — 每次门禁必须验证）

```
1. 列出所有hooks: lefthook.yml → 每个hook的run命令
2. 检查覆盖: gitleaks扫staged还是全量？semgrep扫所有文件还是部分？
3. 验证: 故意引入每个hook应拦截的缺陷→hook拦截了吗？
4. 修复: pre-commit(staged快扫) + CI(full scan) = 分层防线
```

## 中止条件（Quality Gate自身的中止——来自混沌精华）

```
任何检查连续3次FAIL→STOP→不继续后续检查→人工介入
假阳性率>30%→规则需要重写→本次门禁标PARTIAL
```

## 可成长性

```
1. 有检查漏过的缺陷吗？→ 调权重/加检查
2. 假阳性率>30%？→ 重写规则
3. 中止条件触发了吗？→ 调敏感度
→ 轮回采集→确认→注入
```
