# 缉凶 v3.2 — Bug排查合同框架

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 查代码+配置+skill文本的bug。合同链强制执行：每步产出=下步门票。
**Iron Law：** NO FIX WITHOUT A FAILING TEST FIRST. 不复现不修。不CF不提交。
**红线：** 不复现不修。不Counterfactual不提交。不测试不修。3+失败→回分类步重判。不许跳过合同链任何一步。测试必须看它FAIL再修——不是写个测试看它PASS。
**触发：** bug / fix / error / crash / broken / 报错 / 不工作 / not working / bug report / 查skill
**边界：** 功能正确性→缉凶。性能→profiler。安全→铁壁。测试生成→试金石。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | "没报错→T3" | 跳过T1/T2/T5排除 | T3是排除性诊断先排前5个Type |
| 2 | 找到第一个bug就修 | bug叠多层 | 修完继续问"还有没有更深的？" |
| 3 | "偶尔"被忽略 | 间歇≠稳定 | T1:同一输入不同时间不同结果 |
| 4 | campus_check过了=安全 | Masking detection | 手动复现一次 |
| 5 | print让bug消失了 | 观测效应 | print不是修复 |
| 6 | CF写"修后OK"无数据 | 没抓pre/post数值 | CF必须数值对比T3:pre=7.0post=10.0 |
| 7 | 测试一写就PASS | 没看它FAIL就修了 | RED强制:先看FAIL→确认失败原因对→再修 |
| 8 | "测试太复杂写不了" | 设计有问题 | 简化接口再测不硬写复杂测试 |
| 9 | T4回归排查卡在应用层 | 没查nginx/配置/部署变更 | T4首查: git log→配置diff→部署历史→网络层(端口/nginx/proxy_pass)→服务状态→才到代码 |
| 10 | T7被当万能兜底 | 40% agent误判T7(benchmark实测) | T7最罕见——只有代码故意反着需求写才用。不确定→默认T0-T6 |

## Red Lines + Gate Functions（新增——来自测试精华）

| # | 必须 | 证明 | Gate Function |
|---|------|------|---------------|
| 0 | T3门禁 | 引用≥2处原文排除T1/T2/T5 | "我是否真排除了T1(间歇)T2(输入相关)T5(卡住)？"→任一不确定→不判T3 |
| 1 | 先复现再修 | ≥1次稳定复现 | "我是看到了复现还是推测能复现？"→推测→回证据步 |
| 2 | 先CF再提交 | 主动干预+重新观测+T3数值对比 | "CF是改了代码+重新观测还是纯代码对比？"→纯对比不算 |
| 3 | **先写能复现bug的测试再修** | **FAIL→PASS(必须看它先FAIL)** | **"测试第一次跑就PASS？"→没测到bug→回证据步** |
| 4 | 3次修不好→STOP | 3次记录+重分类 | "是否同一策略试了3次？"→换策略 |
| 5 | 修前确认部署映射 | 文件路径+映射 | "生产跑的是哪个文件？"→不确定→读deploy脚本 |
| 6 | 第一个错误别停 | 问"还有没有更深？" | "这个bug是根因还是症状？"→不确定→继续追 |
| 7 | T4先查基础设施层 | git log + config diff + deploy history + 端口/nginx/服务状态 | "基础设施(git/配置/部署/nginx/端口/服务)全查过了？"→任一未查→不进代码层 |

## 合同链

```
[分类]→[证据]→[追踪]→[分析]→[修复]→[验证]→[记录]→[成长]
上一步产出填完整→下一步才能开始。任一步为空→后续无效。
```

详细分类+误判模式+T4/T6强制检查→ [references/bug-classification.md](references/bug-classification.md)

## 可成长性

每次修复完成后强制执行：
```
1. 合同未覆盖的误判模式？→ .fixes/{date}-pattern-candidate.md
2. agent不知道的常识？→ Gotchas候选
3. Gate Function有没有拦住真的误判？→ 验证有效性
4. 同Type+Root模式3+次？→ bug-patterns.md追加pattern
→ 轮回采集→确认→注入
```
