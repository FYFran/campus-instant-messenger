# 缉凶 — 第一性原理 Bug 排查

## CONSTITUTION（不可被 skill-lab 编辑）

**核心功能：** 7-Phase Bug 排查：Classify→Search→Trace→Analyze→Fix→Verify→Grow。不修症状。
**安全约束：** 绝不未复现就修。3+失败=质疑架构。绝不只修一个后端。绝不跳过 Counterfactual。
**触发：** bug/fix/error/crash/broken/报错/不工作/not working/bug report | **边界：** 功能正确性。性能→profiler。安全→铁壁。
---

## Bug 分类学（Phase 1 必查：先分类，后动手）

| Type | 名称 | 判定 | 策略 |
|------|------|------|------|
| **T0** | Deterministic | 固定输入→固定输出，稳定复现 | 标准 7-Phase |
| **T1** | Heisenbug | 加 print 消失/行为变、竞态、时序依赖、死锁/挂起 | 低开销观测 + ⚠️观测者效应警告 + Replay |
| **T2** | Multi-Cause | ≥2 独立因素共同触发 | 每因素独立 Counterfactual |
| **T3** | Silent | 无 crash/error/stack trace，数据/状态悄悄错 | 不变量推断 + 数据 diff + 中间断言 |
| **T4** | Regression | 昨天/上个版本还好好的 | git bisect 定位引入 commit |
| **T5** | State-Machine | 非法状态转移，正常输入→异常输出 | 状态日志→预期转移表→找非法边 |
| **T6** | Environmental | 特定 OS/依赖版本/配置才触发 | 环境快照 + 配置 diff |
| **T7** | Specification | 代码对，需求错 | 不修代码，澄清需求。标记 NOT_A_BUG |

不确定→从 T0 开始，Phase 2 收集中重分类。🔴 **CHECKPOINT:** 分类必须附 1 句依据。

**T3/T5/T6 操作化：**
| T3 | `python -c "print(repr(obj.field))"` 变更前后对比；`diff <(python q_before.py) <(python q_after.py)` |
| T5 | `print(f"[STATE] {old}→{event}→{new}")` 每转移点；列合法转移表→找非法边 |
| T6 | `pip freeze > env.txt; go version; flutter --version`；`diff .env .env.prod`；容器→`docker inspect` |

## Error Recovery

| 类型 | 判定 | 处理 |
|------|------|------|
| **Transient** | 网络超时/429/文件瞬锁 | 3 retry, backoff 1s→4s→16s |
| **Recoverable** | 工具不可用/索引过期 | 降级 TIER1，标记 PARTIAL |
| **Fatal** | 源码损坏/磁盘满/编译不可恢复 | **BLOCK immediately** |

读引入 commit diff → 理解改动意图。非 T4 → 跳过。

### 2c — 收集证据

🔴 **CHECKPOINT:** 无法复现→BLOCK。T6→环境diff。T4→先跑 bisect。环境缺失→PARTIAL。

## Phase 3: Trace

### 3a — 静态追踪
**TIER1:** `rg -n "<function|endpoint>" -t py -t go -t dart` → Read → 手动追踪调用链
**TIER2:** codegraph_context → codegraph_trace → codegraph_callers
🔴 **CHECKPOINT:** 数据流未完→回 3a。

### 3b — Runtime Observe（T0/T1/T2/T5 必跑）
> **原则 (Syncause 83.4%): "Don't guess. Observe."**

⚠️ **观测者效应:** 加观测可能改变时序。T1 加 print 后消失→**诊断已出**—时序敏感，报告。

**TIER1:** 注入到关键函数入口/出口/分支：
```python
print(f"[TRACE] func: var1={var1}, var2={var2}", flush=True)
```
生产/logger过滤/stdout重定向→`logging.debug()` 或 `open('/tmp/trace.log','a').write(...)`。
二进制/bare-metal→`strace -e trace=write -p <pid>` 或 `ltrace`。
死锁/挂起→`pstack <pid>`/`kill -QUIT <pid>`(Java)/`gdb -p <pid> -batch -ex "thread apply all bt"` 抓所有线程栈。
**TIER2:** eBPF (Linux 4.x+) 零开销内核探测。

目标：找到 actual ≠ expected 的精确行。
🔴 **CHECKPOINT:** T0/T1/T2/T5 但未做 Runtime Observe→BLOCK。

### 3c — 双后端检查
| 后端 | 文件 | 部署 |
|------|------|------|
| Python | `campus_app/server/main_remote.py` | → `/app/main.py` |
| Go | `campus_go/main.go` + `handlers/` | 建设中 |

**TIER1:** `rg -n "<bug_pattern>" campus_go/ -t go` 交叉搜。`deploy.py` 确认映射。

## Phase 4: Analyze

### 4a — Divergence Point (TraceFixer)
Runtime Observe 中 expected≠actual 的第一个点。`Expected {X} at L{N}. Actual {Y}.` 根因在**上游**。

### 4b — Delta Debugging (Zeller)
二分隔离：将代码路径/输入/状态空间对半分→含 bug 的那半继续分→最小化到 1 个变更或条件。
格式：`[Hypothesis] bug 在 {X} | [Test] comment/skip {X} → reproduce? | [Result] confirm/reject → next split`
铁律：一次只改一个维度。改两个→不知道哪个引起的。

### 4c — Causal Chain
`{Root} → {Intermediate} → {Divergence} → {Symptom}`

### 4d — Counterfactual (AID)
"修 {root} 后 {symptom} 还出现？" 不会→根因正确。会→还有因素。**T2 每因素独立做。**
🔴 **CHECKPOINT:** 有假设未做 CF→BLOCK。T2 单因素宣布完成→BLOCK。

### 4e — Intermediate Assertion + Hypothesis Log (SpecTune)
关键入口/出口：变量应=X，实际=Y？断言失败→根因在上游。
每轮记录：`Round {n} | Hypothesis: {根因} | Test: {实验} | Result: confirm/reject | Confidence: X.X`
Conf<0.8→SUSPECT。T2 所有因素独立记录。

---

## Phase 5: Fix

**Test Before Fix (Karpathy):** 先写复现测试→确认 FAIL→修复→测试 PASS。测试是真相——测试不过的修复不是修复。

手术式修改 + Critic Auto-Fix Loop:
```
1. Critic → 审查（崩溃/简化/安全/边界/竞态）
2. "没问题"=失职 → 换角度重跑
3. Blocking → auto-fix → re-Critic, max 2 attempts
4. 2次仍 blocking → BLOCK，上报用户+方案
```

🔴 **CHECKPOINT:** >50行→确认。3+失败→Iron Law。T2 声明所有已修+未修因素。未先写测试→BLOCK。

---

## Phase 6: Verify

```bash
python campus_check.py 2>&1 | tee pre-fix.log
# Fix applied...
python campus_check.py 2>&1 | tee post-fix.log
diff pre-fix.log post-fix.log          # 0 新增失败 = OK
rg -n "<bug_pattern>" -t py -t go -t dart
git diff --stat
```

**Masking Detection:** bug 仍在但 check 通过→掩蔽。新 bug 出现→掩蔽确认。回 Phase 4。
🔴 **CHECKPOINT:** 新增失败→回 Phase 5。T2 所有反事实不触发→DONE。

3. 新坑→🔴 CHECKPOINT: "已在 Gotchas？" 不在→追加

## Iron Law

```
NO FIX WITHOUT ROOT CAUSE + COUNTERFACTUAL.
3+ fixes fail → STOP. Question architecture.
T2: fix ONE factor without counterfactual = waste.
```

## Red Flags（任一→停，回 Phase 2）

- "快速修一下"/"试试改 X"/"大概知道原因"(未验证)
- 未追踪完数据流 / 未做 Runtime Observe
- "再试一次"(已 2+ 次)
- T2 修一个因素就宣布完成

## Quality Rules

- **Smart Convergence:** 重跑 Phase 2→3→6。两轮零新发现→DONE，最多 3 轮。二轮互补角度：调用链→竞态/时序。T2 每轮一个因素。
- **Confidence:** >0.8→CRITICAL/HIGH。0.5-0.8→SUSPECT。T2 每因素独立置信度。
- **No-Hedge:** 禁止 might/could/maybe/I think/probably。每条 file:line + concrete reason。不确定→SUSPECT。
- **TRUNCATED AT:** <50% Phase→PARTIAL，quick 豁免。跳过 Runtime Observe(T1/T3/T5)→必找不到根因。
