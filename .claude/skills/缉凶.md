# 缉凶 — 第一性原理 Bug 排查

## CONSTITUTION（不可被 skill-lab 编辑）

**核心功能：** Bug 分类→证据→Runtime Observe→Divergence Point→Counterfactual→手术修复→回归→成长。不修症状。
**安全约束：** 绝不未复现就修。3+失败=质疑架构。绝不只修一个后端。绝不跳过 Counterfactual。
**触发：** bug / fix / error / crash / broken / 报错 / 不工作 / not working / bug report
**边界：** 缉凶管功能正确性。性能问题→profiler。安全漏洞→铁壁。

---

## Bug 分类学（Step -1：先分类，后动手）

不同 bug 需要不同策略。分类错误=浪费时间。

| Type | 名称 | 判定特征 | 首选策略 |
|------|------|---------|---------|
| **T0** | Deterministic | 固定输入→固定输出，稳定复现 | 标准 7-step |
| **T1** | Heisenbug | 加 print 就消失/行为改变、竞态、时序依赖 | 低开销观测 + ⚠️ 观测者效应警告 + Replay |
| **T2** | Multi-Cause | ≥2 独立因素共同触发 | 每因素独立 Counterfactual，逐个排除 |
| **T3** | Silent | 无 crash/error/stack trace，数据或状态悄悄错 | 不变量推断 + 数据 diff + 中间断言 |
| **T4** | Regression | "昨天/上个版本还好好的" | git bisect 定位引入 commit |
| **T5** | State-Machine | 非法状态转移，正常输入→异常输出 | 状态转移追踪，找非法转移边 |
| **T6** | Environmental | 特定 OS/依赖版本/配置才触发 | 环境 diff (版本/配置/文件) |
| **T7** | Specification | 代码逻辑对，需求理解错 | 不修代码，澄清需求。标记 NOT_A_BUG |

如果无法确定类型 → 从 T0 开始，证据收集中重新分类。

## Error Recovery

| 类型 | 判定 | 处理 |
|------|------|------|
| **Transient** | 网络超时/429/文件瞬锁 | 3 retry, backoff 1s→4s→16s |
| **Recoverable** | 工具不可用/索引过期 | 降级 TIER1，标记 PARTIAL |
| **Fatal** | 源码损坏/磁盘满/编译不可恢复 | **BLOCK immediately** |

## Process

### Step -1 — 分类（不可跳过）

读 Bug 分类学表 → 判定 Type 0-7 → 声明分类及依据。
不确定→从 T0 开始，Step 1 收集中重新分类。

### Step 0 — 三源搜索

**TIER1:** 读 `bug-patterns.md` + 项目内搜同模式
```bash
rg -n "<keyword>" --type-add 'code:*.{py,go,dart}' -t code
```
**TIER2:** firecrawl_search + firecrawl_research_search_papers
门控：≥1 来源有结果 → CONTINUE。

### Step 0.5 — Git Bisect（仅 T4 Regression）

```bash
git log --oneline -20                    # 找可疑 commit 范围
git bisect start HEAD <last_known_good>  # 二分定位
# 每个 bisect step: 复现 bug → git bisect bad/good
git bisect reset                         # 结束后重置
```
定位到引入 commit → 读 diff → 理解改动意图 → 继续 Step 1。
不是 T4 → 跳过此步。

### Step 1 — 收集证据

**TIER1:** `python campus_check.py 2>&1` + `curl -s http://139.196.50.134/api/health`
**TIER2:** `ssh root@139.196.50.134 "journalctl -u campus-app --no-pager -n 100"` + audit/nginx 日志

🔴 **CHECKPOINT:** 无法复现 → **BLOCK**。Type-6(环境)→对比环境差异。T4→先跑 git bisect。

### Step 2 — 追踪代码路径

**TIER1 (Static):** `rg -n "<function|endpoint>" -t py -t go -t dart` → Read → 手动追踪调用链
**TIER2:** codegraph_context → codegraph_trace → codegraph_callers

🔴 **CHECKPOINT:** 未追踪完数据流 → 回 Step 2。

### Step 2.5 — Runtime Observe（T0/T1/T2/T5 必跑）

**核心原则（Syncause 83.4% SWE-bench）："Don't guess. Observe."**

⚠️ **观测者效应警告：** 加观测代码可能改变时序。T1 类 bug 如加 print 后消失 → **这就是诊断结果** — bug 对时序敏感。

**TIER1:** 注入 print/log 到关键函数入口/出口/分支点 → 捕获运行时值：
```python
print(f"[TRACE] func_name: var1={var1}, var2={var2}", flush=True)
```
**TIER2:** strace/dtrace 低开销追踪；eBPF (Linux 4.x+) 零开销内核探测。

目标：找到 actual behavior 偏离 expected behavior 的精确行。

### Step 3 — 双后端检查

| 后端 | 关键文件 | 部署 |
|------|---------|------|
| Python | `campus_app/server/main_remote.py` | → `/app/main.py` |
| Go | `campus_go/main.go` + `handlers/` | 建设中 |

**TIER1:** `rg -n "<bug_pattern>" campus_go/ -t go` 交叉搜。
门控：两个后端都检查过 → PASS。

### Step 4 — Divergence Point + Counterfactual

缉凶核心。不是"Trace→Check"，是分叉点定位 + 反事实排除多因。

1. **Divergence Point (TraceFixer):** Runtime Observe 中 expected ≠ actual 的第一个点。
   格式：`Expected {X} at line {N}. Actual {Y}.` 根因在此点**上游**。
2. **Causal Chain:** `{Root} → {Intermediate} → {Divergence} → {Symptom}`
3. **Counterfactual (AID):** "如果修了 {root}，{symptom} 还会出现？" 不会→根因正确。会→还有别的因素。T2 每因素独立 Counterfactual。
4. **Intermediate Assertion (SpecTune):** 关键函数入口/出口：变量应=X，实际=Y？断言失败→根因在上游。
5. **Conclusion:** `{机制} → {因果链} → {症状} (confidence: X.X)`。Confidence<0.8→SUSPECT。T2 标记所有因素。

### Step 5 — 修复 + Critic Auto-Fix

手术式修改 + Critic Auto-Fix Loop:
```
1. Critic → 审查（崩溃点/简化/安全/边界/竞态）
2. "没问题" = 失职 → 换角度重跑
3. Blocking issues → auto-fix → re-Critic, max 2 attempts
4. 2次仍 blocking → BLOCK，上报用户+方案
```

🔴 **CHECKPOINT:** >50 行→确认。3+ 失败→Iron Law。T2 必须声明所有已修+未修因素。

### Step 6 — 回归验证 + Masking Detection

```bash
python campus_check.py 2>&1 | tee pre-fix.log
# Fix applied...
python campus_check.py 2>&1 | tee post-fix.log
diff pre-fix.log post-fix.log          # 0 新增失败 = OK
rg -n "<bug_pattern>" -t py -t go -t dart  # 搜同模式
git diff --stat                         # 确认改动范围
```

**Masking Detection:** bug 仍在但 campus_check 通过 → 掩蔽。新 bug 出现 → 掩蔽确认。回 Step 4。

🔴 **CHECKPOINT:** diff 有新增失败 → 回 Step 5。T2 所有反事实都不触发 → 可标记 DONE。

### Step 7 — 报告 + 成长

```markdown
## Bug Report — {title}
**Type:** T{n} | **Steps:** {n}/7 | **Mode:** {TIER1/TIER2}
**Divergence:** Expected {X} at line {N}. Actual {Y}.
**Causal Chain:** {Root} → {Intermediate} → {Divergence} → {Symptom}
**Root Cause:** ... (confidence: X.X)
**Counterfactual:** 修 {root} 后 bug 还出现？→ {是/否}
**Fix:** {file:line-range} — {改动}
**Verified:** {check/flutter/manual} | **Regression:** {n} instances
**TRUNCATED:** {跳过原因。quick豁免}
```
产物 → `.fixes/{date}-{slug}.md`（`.gitignore` 此目录）。

| CHECKPOINT | 条件 | 动作 |
|-----------|------|------|

## Iron Law

```
NO FIX WITHOUT ROOT CAUSE + COUNTERFACTUAL.
3+ fixes fail → STOP. Question architecture.
T2 bugs: fix ONE factor without counterfactual = waste.
```

## Red Flags（任一 → 停，回 Step 1）

- "快速修一下"/"试试改 X"/"大概知道原因"(未验证)
- 未追踪完数据流 / 未做 Runtime Observe
- "再试一次"(已 2+ 次)
- T2 bug 只修一个因素就宣布完成

## Quality Rules

- **Smart Convergence:** 修复后重跑 Step 1→2→2.5→6。两轮零新发现 → DONE。最多 3 轮。第二轮从互补角度：首轮追调用链→次轮追竞态/时序。T2→每轮追一个因素。
- **Confidence:** >0.8 → CRITICAL/HIGH。0.5-0.8 → SUSPECT。T2 每因素独立置信度。
- **No-Hedge:** 禁止 might/could/maybe/I think/probably。每条 finding: file:line + concrete reason。
- **TRUNCATED AT:** <50% steps → PARTIAL。quick 模式豁免。
- **Observability Ceiling:** 静态分析看不到的 bug (T1/T3/T5) → Runtime Observe 必须跑。跳过=必然找不到根因。
