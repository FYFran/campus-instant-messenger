# 缉凶 v2.0 — 合同框架

## CONSTITUTION（不可被 skill-lab 编辑）

**核心功能：** 查代码+配置+skill 文本的 bug。合同链强制执行：每步产出=下步门票。
**安全约束：** 不复现不修。不 Counterfactual 不提交。不测试不修。3+失败→质疑架构。
**触发：** bug / fix / error / crash / broken / 报错 / 不工作 / not working / bug report / 查skill
**边界：** 功能正确性→缉凶。性能→profiler。安全→铁壁。

---

## Red Lines（每条修复必须满足）

| # | 必须做的事 | 做完的证明 |
|---|-----------|----------|
| 1 | **先复现再修** — 至少 1 次稳定复现 | 复现步骤记录 + baseline 输出 |
| 2 | **先 Counterfactual 再提交** — "修了 X, bug 不再出现"要有 pre/post 对比数据 | CF 含具体数字/日志/测试结果 |
| 3 | **先写能复现 bug 的测试再修** — 测试先 FAIL, 修复后 PASS | FAIL→PASS 完整输出 |
| 4 | **3 次修不好 → STOP, 回分类步重判** — 根因可能判错了 | 3 次记录 + 重分类依据 |
| 5 | **修前读 `deploy.py`** — 确认哪个文件跑在生产上 | 文件路径 + 部署映射确认 |

---

## 合同链（每步产出 = 下一步门票）

```
[分类] → [证据] → [追踪] → [分析] → [修复] → [验证] → [记录]

规则：上一步产出填完整 → 下一步才能开始。
      任一步产出为空 → 后续全部无效。
      事后补填 → 标记 POST-HOC，置信度降一级。
```

| 步 | 产出（必须填写） | 未填写的后果 |
|----|----------------|-------------|
| **分类** | `Type: T___ 依据: ___` | 后续搜索无目标 |
| **证据** | `复现步骤: ___` + baseline 输出 | 追踪无起点 |
| **追踪** | `调用链: ___` + `Expected vs Actual 第一点: ___` | 分析无数据 |
| **分析** | `根因: ___ (conf: ___)` + Counterfactual 结果。Conf<0.8→SUSPECT。3+次修复失败→STOP，回分类步重判 | 修复无依据 |
| **修复** | 测试 FAIL→PASS 完整输出。修前确认 `deploy.py` 映射 | 验证无基线 |
| **验证** | pre/post diff + `git diff --stat` | 记录不完整 |
| **记录** | `.fixes/{date}-{slug}.md` + bug-patterns 更新 | 经验丢失 |

## Bug 分类（先看症状→选 Type→给证据）

分类不是猜。症状决定 Type，证据证明 Type。每条分类必须附一句"为什么是 T_X 而不是 T_Y"。

**决策流：**

```
1. 代码没改但昨天能用？           → T4。立刻查 git log + 配置变更
2. 每次必现、稳定复现？            → T0。直接走合同链
3. 有时候出现、加 log 就消失？     → T1。竞态/时序/Heisenbug
4. 出现需要 ≥2 个条件同时满足？     → T2。每因素独立验证
5. 正常输入→卡在中间状态回不来？   → T5。状态机卡住。查状态转移日志
6. 没报错没 crash，就是结果不对？   → T3。数据悄悄错/性能退化/误算。T5已排除
7. 只在本机/CI/Docker/某 OS 出现？  → T6。环境快照对比
8. 读代码发现行为符合产品设计？     → T7。STOP。NOT_A_BUG
```

| Type | 症状关键词 | 第一步动作 | 常见误判 |
|------|-----------|---------|---------|
| **T0** | "每次都"、"必现"、"稳定复现" | 标准合同链 | — |
| **T1** | "偶尔"、"有时候"、"加 print 好了"、"并发" | 只读观测（不加 print），时序分析 | 误判为 T0 — 间歇≠稳定 |
| **T2** | "同时满足"、"和...有关"、"跟...也有关" | 每因素独立 CF，修一个不收工 | 误判为 T0 — 多因素≠单因素 |
| **T3** | "没报错"、"数字不对"、"越来越慢"、"悄悄错" | 不变量检查 + 数据 diff | 误判为 T0 — 无 crash≠简单 bug |
| **T4** | "昨天还好"、"代码没改"、"重启后"、"升级后" | git bisect + 配置 diff + 时间线 | 误判为代码 bug — 先查环境 |
| **T5** | "卡住了"、"状态不变"、"一直 pending" | 状态转移日志 + 状态机图 | 误判为 T0(卡住≠crash) 或 T3(卡住也"无报错结果不对"，先问 T5) |
| **T6** | "本机正常 CI 挂了"、"Windows 行 Linux 不行" | 环境快照 diff（OS/Go版本/Docker） | 误判为 T0 — 本机能跑≠不是 bug |
| **T7** | "以为"、"应该是"、"别人都能"、读代码发现需求如此 | STOP。不修代码。出产品建议 | 误判为 T0-T6 — 需求≠代码错误 |

**分类不自洽 → 回第一步重判。** 比如分类选了 T3 但说"报错了"→矛盾→重判。

## MEL 模式

| 模式 | 合同链 | 触发 |
|------|--------|------|
| **full** | 7 步全走 | 默认 |
| **quick** | 分类+证据+分析（不修） | 快速/诊断/看看 |
| **safe** | 7 步 + Phase 5 [AWAIT APPROVAL] | 重要修复 |
| **emergency** | 只 Red Line 1-3。事后 24h 补链 | 生产挂了/紧急/立刻 |

## Gotchas（必须做的正向检查）

| # | 场景 | 正向动作 |
|---|------|---------|
| 1 | 排查到根因了 | 标 SUSPECT（除非 conf≥0.8）。不确定=不确定 |
| 2 | T1 加 print bug 消失了 | 诊断到此为止。print 只是观测工具，不是修复 |
| 3 | T2 修了一个因素 | 继续修下一个因素。T2 必须所有因素都独立 CF 通过 |
| 4 | 修完了一个 bug | rg 全库搜同模式。一个地方有 bug→其他地方可能也有 |
| 5 | campus_check 过了 | 手动复现一次。check 可能漏（Masking detection） |
| 6 | 临时文件写在哪 | 当前目录。Win 无 /tmp |
| 7 | T3 静默数据错误 | CF 必须含具体数值对比（修前=7.0 vs 修后=10.0）。只说"功能正常了"=TEMPLATE，无效 |
| 8 | CF 写了"修后 OK"但无数字 | 回到证据步，抓 pre/post 数据。CF 没数据=没验证 |

## Worked Examples

**T0 — 稳定复现:**
```
Bug: 活动列表 API 返回 500
→ 分类: T0 (每次空数据库必 500，稳定复现)
→ 证据: curl /api/activities → 500; campus_check baseline captured
→ 追踪: ListActivities→db.Query→rows.Scan at activities.go:45
         Expected: rows.Err() check. Actual: nil deref (rows=nil)
→ 分析: rows.Err() 缺失 (conf:0.95)。CF: 加检查→curl→200 OK
→ 修复: go test→FAIL→加 rows.Err()→PASS
```

**T1 — 竞态/时序:**
```
Bug: 同一个学生偶尔报了两次名
→ 分类: T1 (每20-30次一次，加log频率降低 → 观测者效应)
→ 证据: 10并发→2条重复。时序分析: SELECT和INSERT之间有窗口
→ 追踪: Signup→SELECT COUNT→INSERT at activities.go:156,205
         Expected: 第二次SELECT看到第一次的INSERT。Actual: 两次都看到0
→ 分析: SELECT+INSERT TOCTOU窗口 (conf:0.85)。CF: 加FOR UPDATE→0重复
→ 修复: FOR UPDATE + ON CONFLICT + UNIQUE → 并发测试PASS
```

**T3 — 静默数据错误:**
```
Bug: 学生志愿时长统计显示 7 小时，实际签到 10 小时
→ 分类: T3 (没报错没crash，结果不对。先排除了 T5——状态转移正常)
→ 证据: GET /api/my-stats → {hours:7}。数据库 SUM=10。差 3 小时。
         基线: 活动A(3.5h)+活动B(2.0h)+活动C(4.5h)=10.0h。API返回7.0。
→ 追踪: my-stats→SUM(hours)→hours字段来自 certificates.hours(FLOAT)
         Expected: sum(3.5,2.0,4.5)=10.0。Actual: API返回7.0。
→ 分析: int() truncation 在累加前发生 (conf:0.92)。
         CF: round()替代int()→相同数据→API返回10.0→差3小时消失。
         修前=7.0, 修后=10.0, Δ=+3.0。
→ 修复: int(h)→round(h) at stats.py:156。test→FAIL(7.0≠10.0)→修→PASS(10.0)
```

**T5 — 状态机卡住:**
```
Bug: 报名后状态一直 pending，永远不变 selected
→ 分类: T5 (正常输入→卡在 pending 回不来。先排除 T3——数据没算错，状态没转移)
→ 证据: POST signup→pending。2h后 GET /my-signups→仍 pending。活动已 ended。
         基线: 同活动 3 个学生都卡 pending，0 个 selected。数据库确认。
→ 追踪: Signup→INSERT status='pending'→(无后续转移代码)
         Expected: deadline 后 pending→selected/waitlist。Actual: 永远 pending。
→ 分析: 状态机缺自动转移路径 (conf:0.95)。Go 后端无抽签/自动结束逻辑。
         CF: 加 lottery handler + 定时器→同活动→pending 自动变 selected/waitlist。
→ 修复: 新增 lottery.go + StartAutoProcessor → test: pending→selected ✓
```

**T6 — 特定环境:**
```
Bug: CI 挂了但本机正常
→ 分类: T6 (Go 1.22本机PASS, Go 1.23 CI panic → 环境差异)
→ 证据: 环境快照: Go 1.22vs1.23, SQLitevsPostgreSQL, NULL处理差异
→ 追踪: ListActivities→rows.Scan→pgx检测sql.Scanner→NULL委托→nil deref
         Expected: NULL→零值。Actual: NULL→panic (Go 1.23改变了Scan行为)
→ 分析: Go 1.23+pgx NULL Scan行为变更 (conf:0.9)。CF: 加COALESCE→CI PASS
→ 修复: COALESCE(a.created_at,'epoch') + sql.NullString for nullable columns
```

**T7 — NOT_A_BUG:**
```
Bug: "我的报名"只显示已确认的
→ 分类: T7 (读代码发现 WHERE signup_status='confirmed' 是产品设计)
→ 证据: GetMySignups SQL 明确过滤 confirmed — 设计如此，不是 bug
→ 追踪: GetMySignups→JOIN signups→WHERE status='confirmed'
→ 分析: NOT_A_BUG。正确响应: STOP，不修代码。
         产品建议: 前端加"待审批报名在'我的申请'页面查看"的提示
→ 修复: 无代码修改
```
