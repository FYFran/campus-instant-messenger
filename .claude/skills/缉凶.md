# 缉凶 v2.6 — 合同框架 (D2 三叉戟: 类型门禁 + 症状锚定 + T0/T3边界)

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 查代码+配置+skill 文本的 bug。合同链强制执行：每步产出=下步门票。
**安全约束：** 不复现不修。不 Counterfactual 不提交。不测试不修。3+失败→质疑架构。
**触发：** bug / fix / error / crash / broken / 报错 / 不工作 / not working / bug report / 查skill
**边界：** 功能正确性→缉凶。性能→profiler。安全→铁壁。

## Red Lines（每条修复必须满足）

| # | 必须做的事 | 做完的证明 |
|---|-----------|----------|
| 0 | **T3 门禁** — 选 T3 前必须引用 bug 描述中 ≥2 个具体词句，分别证明 T1/T2/T5 不成立。缺一条→不能选 T3 | 三条排除各引用 bug 描述原文 |
| 1 | **先复现再修** — 至少 1 次稳定复现 | 复现步骤记录 + baseline 输出 |
| 2 | **先 Counterfactual 再提交** — "修了 X, bug 不再出现"要有 pre/post 对比数据 | CF 含具体数字/日志/测试结果 |
| 3 | **先写能复现 bug 的测试再修** — 测试先 FAIL, 修复后 PASS | FAIL→PASS 完整输出 |
| 4 | **3 次修不好 → STOP, 回分类步重判** — 根因可能判错了 | 3 次记录 + 重分类依据 |
| 5 | **修前读 `deploy.py`** — 确认哪个文件跑在生产上 | 文件路径 + 部署映射确认 |
| 6 | **找到第一个错误别停** — 编译/语法错误修了 bug 还在？继续往下挖。bug 常叠多层 | 至少问一次 "还有没有更深的？" |

## 合同链（每步产出 = 下一步门票）

```
[分类] → [证据] → [追踪] → [分析] → [修复] → [验证] → [记录]

规则：上一步产出填完整 → 下一步才能开始。
      任一步产出为空 → 后续全部无效。
      事后补填 → 标记 POST-HOC，置信度降一级。
```

| 步 | 产出（必须填写） | 未填写的后果 |
|----|----------------|-------------|
| **分类** | `Type: T___ 依据: ___` | 后续搜索无目标 |
| **证据** | `复现步骤: ___` + baseline 输出 | 追踪无起点 |
| **追踪** | `调用链: ___` + `Expected vs Actual 第一点: ___` | 分析无数据 |
| **分析** | `根因: ___ (conf: ___)` + Counterfactual 结果。Conf<0.8→SUSPECT。3+次修复失败→STOP，回分类步重判 | 修复无依据 |
| **修复** | 测试 FAIL→PASS 完整输出。修前确认 `deploy.py` 映射 | 验证无基线 |
| **验证** | pre/post diff + `git diff --stat` | 记录不完整 |
| **记录** | `.fixes/{date}-{slug}.md` + bug-patterns 更新 | 经验丢失 |

🛑 **症状锚定检查 — 追踪步产出填完后、分析步开始前强制执行：**

1. 我追踪到的根因 → 产生的症状是: ___
2. 原 bug 报告描述的症状是: ___
3. 两者一致？
   - ✅ → 进入分析步
   - ❌ → 这不是原报告的 bug。记录 `Bonus Finding: ___`（找到的意外 bug），回到证据步用排除法重追原症状。

不填此检查 → 分析步产出无效。

## Bug 分类（先看症状→选 Type→给证据）

分类不是猜。症状决定 Type，证据证明 Type。每条分类必须附一句"为什么是 T_X 而不是 T_Y"。

**分类前强制停顿：**
```
STOP. 读 bug 描述 2 遍。问自己:
  "这个 bug 的深层因果结构是什么？"
  "什么变了？什么时候变？谁受影响？"

不匹配关键词。想 failure mode。
```

**决策流：**

```
1. 代码没改但昨天能用？           → T4。立刻查 git log + 配置变更
2. 每次必现 crash/panic/500/nil deref/SIGSEGV → T0。必须有崩溃证据(stack trace / panic log / exit code≠0)。无报错→不是 T0。**判别铁律：看用户症状（用户看到了crash/500吗？），不看代码内部（代码里有nil pointer≠用户看到crash）。代码内部nil+用户无crash报告 → 不是T0**。不是"每次都慢"——那是 T3
3. 同一操作，有时正常有时异常？     → T1。同一输入不同时间=不同结果。时序/竞态/Heisenbug
4. 换一个参数值（学院名/ID）就正常？ → T2。不同输入=不同结果。区分:改输入能改变结果吗？能→T2，不能→T1
5. 正常输入→卡在中间状态回不来？   → T5。状态机卡住。查状态转移日志
6. 1-5全排除了？                   → T3。跳过1-5直接选T3=整条链报废。T3要求每次结果一致地错
7. 只在本机/CI/Docker/某 OS 出现？  → T6。环境快照对比
8. 读代码发现行为符合产品设计？     → T7。STOP。NOT_A_BUG
9. 症状可被部署配置差异(nginx/proxy_pass/port/env)解释**且近期有部署/重启**？ → T4 (NOT 代码 bug)。⚠ T4 专属配置检查(F6)通过前不许读源码
```

| Type | 症状关键词 | 第一步动作 | 常见误判 |
|------|-----------|---------|---------|
| **T0** | "每次都"、"必现"、"稳定复现" | 标准合同链 | — |
| **T1** | "偶尔"、"有时候"、"刷新又好了"、"不是每次都"、"并发" | 只读观测（不加 print），时序分析 | 误判为 T0(间歇≠稳定) 或 T3(间歇也"无报错"，先问 "刷新变化吗?") |
| **T2** | "同时满足"、"和...有关"、"跟...也有关"、"某些...某些不" | 每因素独立 CF，修一个不收工 | 误判为 T0(不是crash) 或 T3(T2也"没报错不同输入不同结果"，先问"换个参数值呢?") |
| **T3** | "没报错"、"数字不对"、"越来越慢"、"悄悄错" | 不变量检查 + 数据 diff。先排 T1(刷新变化?) T2(输入相关?) T5(卡住?) | 误判为 T0(无crash≠简单) 或来自 T1/T2(先排它们再判T3) |
| **T4** | "昨天还好"、"代码没改"、"重启后"、"升级后" | git bisect + 配置 diff + 时间线。⚠ T4 专属配置检查(F6)通过前不许读源码 | 误判为代码 bug — 先查环境 |
| **T5** | "卡住了"、"状态不变"、"一直 pending" | 状态转移日志 + 状态机图 | 误判为 T0(卡住≠crash) 或 T3(卡住也"无报错结果不对"，先问 T5) |
| **T6** | "本机正常 CI 挂了"、"Windows 行 Linux 不行" | 环境快照 diff（OS/Go版本/Docker） | 误判为 T0 — 本机能跑≠不是 bug |
| **T7** | "以为"、"应该是"、"别人都能"、"安全审查说"、"OWASP推荐"、读代码发现需求如此 | STOP。第三方报告≠bug——先验证声称的问题是否真实存在。不修代码。出产品建议 | 误判为 T0-T6 — 需求/报告≠代码错误。安全审查结论也需独立判断 |

**分类不自洽 → 回第一步重判。** 比如分类选了 T3 但说"报错了"→矛盾→重判。

🛑 **类型门禁：** 分类必须从 T0-T7 中选。T_AUTH / T_TOKEN / T9 等自创类型 → 分类无效，回第一步重判。如果 T0-T7 装不下 → 你找到的是意外 bug 不是原报告症状 → 记录 `Bonus Finding`，回分类步按原症状重判。

**致命误判模式（从真实失败中提取。每条 = 一个被误判的 bug）：**
| # | 模式 | 症状信号 | 为什么不是 T3 | 真实 Type |
|---|------|---------|-------------|----------|
| F1 | "偶尔不生效"+"刷新又好了" | 同一操作结果不一致 | T3要求每次一致地错——这个不一致 | T1 |
| F2 | "某些学院名能看到，某些不能" | 不同输入不同结果 | T3要求与输入无关——这个输入决定结果 | T2 |
| F3 | "一直 pending 不变" | 卡在中间状态 | T3是数据错——这个是状态不转移 | T5 |
| F4 | "偶尔扣钱不给结果"+"约20次1次" | 间歇症状但非竞态 | T1是同一输入不同时间不同结果——这个是逻辑错误(扣费先于调用)间歇触发 | T3 |
| F5 | "安全审查说XX不安全"/"OWASP推荐XX" | 第三方报告声称有bug | T7: 先判断报告声称的问题是否真实存在。安全审查结论≠bug | T7 |
| F6 | "昨天还好+重启后+代码没改"被当代码bug | Agent 立即读源码不查配置 | T4 专属配置检查(见下)通过前不许读源码 | T4 |
| F7 | "T4 + token/401/认证" 被当代码bug | Agent 直接读 auth.go 查 JWT_SECRET | 第一步 curl 直连后端端口(绕过nginx) → 后端正常? → nginx问题。后端也401? → 代码问题。不经过此网络测试不许读源码 | T4 |
| F8 | "我找到了 X 类型的 bug" 但跟原报告症状对不上 | 追踪代码时发现了真实 bug，但分类的是意外发现而非原症状。Agent 自创 T_AUTH / T9 等非 T0-T7 类型，或对意外发现做了正确分类但目标错了 | 症状锚定检查(见合同链)填完才能进分析步。意外 bug 记 Bonus Finding，回证据步重追原症状 | 原症状的真实 Type |

🛑 判 T3 前必须对照此表。命中任一模式→改选对应 Type。
🛑 T1 vs T3 区分铁律: "间歇性症状≠竞态条件。问：是同一输入不同时间不同结果(T1)还是逻辑错误条件触发(T3)？"

**T4 强制配置检查（分类为 T4 时走完这 3 步才能读源代码）：**
```
1. 配置 diff: nginx.conf / docker-compose*.yml / .env / systemd unit / 部署脚本 — 与 git HEAD 对比
2. 端口/服务检查: 确认服务监听端口和 proxy_pass/upstream 目标是否匹配
3. 启动日志: 检查上次重启/部署日志，找 config loaded / reverted / warning / error
```
🛑 这 3 步没走完 → 不许打开任何 .go / .py 源码文件。
✅ 3 步全走过且配置无异常 → F6 门禁解除，正常进入合同链追踪步读源码。
⚠ 第一步 "配置 diff" 范围 = 该项目所有非源码配置文件(nginx/yaml/env/docker/systemd)，不限于单个文件。

**结构性排除（填完才能进证据步）：**
```
最可能的替代分类: T_?（你的分类之外，哪个最容易混淆？）
替代分类的证据: ___（引用 bug 描述中的具体词句）
你的分类 vs 替代的决定性差异: ___（一句话。这不是自评——引用具体证据）
```
🛑 规则: "决定性差异"不能写"I think/believe/认为"——必须是可验证的事实差异。

## Gotchas（必须做的正向检查）

| # | 场景 | 正向动作 |
|---|------|---------|
| 1 | 排查到根因了 | 标 SUSPECT（除非 conf≥0.8）。不确定=不确定 |
| 2 | T1 加 print bug 消失了 | 诊断到此为止。print 只是观测工具，不是修复 |
| 3 | T2 修了一个因素 | 继续修下一个因素。T2 必须所有因素都独立 CF 通过 |
| 4 | 修完了一个 bug | rg 全库搜同模式。一个地方有 bug→其他地方可能也有 |

## Worked Examples
**T0 — 稳定复现:**
```
Bug: 活动列表 API 返回 500
→ 分类: T0 (每次空数据库必 500，稳定复现)
→ 证据: curl /api/activities → 500; campus_check baseline captured
→ 追踪: ListActivities→db.Query→rows.Scan at activities.go:45
         Expected: rows.Err() check. Actual: nil deref (rows=nil)
→ 分析: rows.Err() 缺失 (conf:0.95)。CF: 加检查→curl→200 OK
→ 修复: go test→FAIL→加 rows.Err()→PASS
```

**T1 — 时序/竞态 (对比: 同一 bug，两条路径):**

```
Bug: Python 后端积分更新偶尔不生效。有时刷新还是旧的。再刷一次对了。

❌ 误判路径 (→T3):
→ 分类: T3 (没报错没crash，结果不对)
  致死错误: 只看到"结果不对"→直接选T3→跳过T1检查。"偶尔"被忽略。
  T3要求每次结果一致地错——这个bug每20次正常1次错，"一致性"不成立。

✅ 正确路径 (→T1):
→ 分类: T1 (同一操作不同时间结果不同=时序/竞态)
  关键信号拆解:
  - "偶尔"+"不是每次都"=同一输入结果不一致→T1(不是T3)
  - "再刷一次对了"=观测不改变行为→非Heisenbug子型
  因果结构: coroutine未await→GC随机回收→间歇性不执行
→ 证据: 连续请求10次→2次积分未更新。baseline: 正常10次全更新。
→ 追踪: complete_activity→INSERT certificates→await缺失→coroutine→GC
         Expected: coroutine always runs to completion. Actual: GC回收未await协程
→ 分析: missing await (conf:0.88)。CF: 加await→10次全过。
→ 修复: 加 await at main.py:1095
```

**T2 — 多因素 (对比: B03 college scope):**
```
Bug: college_admin 有时能看到其他学院的活动。跟学院名有关。

❌ 误判路径 (→T3):
→ 分类: T3 (没报错没crash，看到不该看的结果不对)
  致死错误: "结果不对"→直接T3。没问:换个学院名还触发吗?
  T3要求每次结果一致——但这bug只在特定学院名出现。

✅ 正确路径 (→T2):
→ 分类: T2 (某些学院名触发其他不触发→不同输入不同结果)
  区分T1/T2: 改输入(学院名)能改变结果→T2。T1是同一输入不同时间。
  两个因素: (1)学院名含特殊字符 (2)strings.Contains部分匹配→单修一个不够
→ 证据: 学院A"计算机"=正常。学院B"计算机(师范)"=跨院可见。
→ 追踪: college check→strings.Contains→"计算机"匹配"计算机(师范)"
         Expected: 精确匹配。Actual: 子串匹配绕过。
→ 分析: strings.Contains过宽 (conf:0.9)。CF: Split逐项==→不再跨院。
→ 修复: strings.Contains→strings.Split+循环逐一==
```

**T3 — 静默数据错误:**
```
Bug: 学生志愿时长统计显示 7 小时，实际签到 10 小时
→ 分类: T3 (排T1:每次结果一致地错=10.0→7.0=固定偏差; 排T2:所有活动都受影响与参数无关; 排T5:无状态卡住)
→ 证据: GET /api/my-stats → {hours:7}。数据库 SUM=10。差 3 小时。
         基线: 活动A(3.5h)+活动B(2.0h)+活动C(4.5h)=10.0h。API返回7.0。
→ 追踪: my-stats→SUM(hours)→hours字段来自 certificates.hours(FLOAT)
         Expected: sum(3.5,2.0,4.5)=10.0。Actual: API返回7.0。
→ 分析: int() truncation 在累加前发生 (conf:0.92)。
         CF: round()替代int()→相同数据→API返回10.0→差3小时消失。
         修前=7.0, 修后=10.0, Δ=+3.0。
→ 修复: int(h)→round(h) at stats.py:156。test→FAIL(7.0≠10.0)→修→PASS(10.0)
```

**T4 — 回归:**
```
Bug: JWT token 刷新昨天能用今天全401。代码没改，重启过。
→ 分类: T4 (代码没改昨天能用→回归)
→ 证据: git log→3天无变更。重启时间=首次bug时间。JWT_SECRET未加载。
→ 追踪: token/refresh→Getenv("JWT_SECRET")→""。Expected:持久化。Actual:丢失。
→ 分析: JWT_SECRET未持久化 (conf:0.95)。CF:设env→重启→正常。
→ 修复: systemd Environment= 或 /etc/environment
```

**T5 — 状态机卡住:**
```
Bug: 报名后状态一直 pending，永远不变 selected
→ 分类: T5 (正常输入→卡在 pending 回不来。先排除 T3——数据没算错，状态没转移)
→ 证据: POST signup→pending。2h后 GET /my-signups→仍 pending。活动已 ended。
         基线: 同活动 3 个学生都卡 pending，0 个 selected。数据库确认。
→ 追踪: Signup→INSERT status='pending'→(无后续转移代码)
         Expected: deadline 后 pending→selected/waitlist。Actual: 永远 pending。
```
