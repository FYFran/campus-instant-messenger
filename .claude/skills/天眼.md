# 天眼 v1.1 — 可观测性引擎

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 全栈可观测性：指标+日志+追踪→仪表盘→告警→事件响应→事后分析。系统只有在能被理解时才是可靠的。
**Iron Law：** NO ALERT WITHOUT A RUNBOOK. 无runbook的告警=噪声=静默。
**红线：** 绝不告警内部指标(CPU/内存不是用户可见)。绝不接受静态阈值(SLO消耗率优先)。绝不产生未聚合的告警噪声。绝不设>2个SLO/服务起步。绝不接受>99.99%的SLO目标(需执行批准+双倍预算)。绝不对一切告警。
**触发：** 监控 / observability / monitor / 告警 / alerting / SLO / metrics / 看板 / dashboard / 日志 / logging
**边界：** 可观测性→天眼。部署→布阵(P0稳态表征+事后分析→天眼原则)。安全监控→铁壁Phase 3。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | 告警70%是误报 | 静态阈值+无基线 | 基线7天→消耗率窗口→滞后防抖 |
| 2 | 50条同错误告警 | 无聚合 | 聚类错误→1告警+计数 |
| 3 | "CPU>80%告警" | 告警原因非症状 | P95延迟>500ms=告警。CPU>80%=原因 |
| 4 | 99.99% SLO花了3倍成本 | SLO越高成本指数级 | 99.9%够用就不设99.99% |
| 5 | 仪表盘30个面板 | 认知过载 | 7±2面板/屏。F布局 |
| 6 | 可观测性成本>基础设施15% | 全保真度永久保留 | 层级化:7天全→90天降采样→归档 |
| 7 | 告警静默过多 | 已知问题不断触发 | 静默告警30天后自动删除。已知问题→修根因而非加静默规则 |

## 可观测性严重度分级

| 级别 | 定义 | 示例 |
|------|------|------|
| 🔴 CRITICAL | 用户已受影响但无告警 | 生产停机无检测、SLO breach无通知 |
| 🟠 HIGH | 告警存在但无runbook/滞后 | 无runbook告警、静态阈值漂移>30% |
| 🟡 MEDIUM | 仪表盘/数据质量问题 | 面板>15个、降采样丢失关键信号 |
| 🟢 LOW | 优化建议 | 面板布局调整、标签命名统一 |

## 十二条法律（来自5个可观测性来源skill精华）

### 第一条: 三大支柱完整性
```
系统不可被观测 除非: 结构化指标 + 结构化日志 + 分布式追踪
缺二=不可接受。
指标答"什么?" 日志答"为什么?" 追踪答"哪里?"
```

### 第二条: 面向用户的北极星
```
每条指标/告警/面板必须可追溯至用户可见影响
问: "这个变了用户会注意到吗?"
否→实现细节→记录不设SLO
CPU/内存/磁盘/队列深度→内部。P50/P95/P99延迟→面向用户
```

### 第三条: 层级化仪表盘
```
L1(概览): SLO成就+错误预算+活跃告警+业务KPI
L2(服务): 黄金信号(延迟/流量/错误/饱和度)
L3(组件): RED方法(率/错误/持续时间)
L4(实例): USE方法(利用率/饱和度/错误)
7±2面板/屏。每个面板服务一个角色
```

### 第四条: 告警纯度法则
```
每条告警必须同时: 可操作+新颖+具体(含runbook)+用户可见
脚本能处理→自动修复(不告警)
已知问题→静默
无runbook→噪声→删除
用户未受影响→日志不告警
```

### 第五条: SLO治理
```
每服务1-2个SLO起步。目标来自用户需求非技术能力
一般: 99%(~7h/月)
高级: 99.9%(~43min/月)
临界: 99.99%(~4min/月) — 需执行批准+双倍预算
错误预算=用来花的非囤的。季度审查SLO
```

### 第六条: 基线先于阈值
```
无基线不设阈值。无≥7天数据不设基线。
阈值绑定统计偏差非任意值。
每季度审查阈值。
基线自适应: 每周滚动重算
```

### 第七条: 消耗率优于单点
```
多窗口消耗率告警保护SLO:
  快速: 1h内消耗SLO预算2%
  慢速: 3d内消耗SLO预算10%
滞后: 触发阈值≠解决阈值(触发5%→解决3%)
```

### 第八条: 成本意识
```
可观测性成本≤基础设施15%
层级化: 7d全保真→90d降采样→1-7y归档
采样: 高流量→尾部感知采样。错误→全保真
高基数标签→检测→控制
```

### 第九条: 系统性升级
```
每条告警: 严重级别+升级路径+确认时间+runbook
严重: 停机 紧急: 接近阈值 信息: 部署
升级: 5min初级→15min次级→30min经理(线性+时间分段)
```

### 第十条: 主动脉冲
```
每次健康检查: 扫描近15min日志→比较基线→检测新模式→跨服务关联
在告警前发现问题。
部署前负载测试: 最大负载+临界点+恢复时间
```

### 第十一条: 事后学习
```
每次事件产出: 更正runbook+更新阈值+SLO审查+预防措施
24h内事后总结会议
归因系统不归因人
事件→CI/CD门禁防止重复
```

### 第十二条: 金发姑娘数据策略
```
边缘: 每节点≤1h全保真
主干: 5s聚合指标+关联ID日志
短存: 7d全保真指标+日志
长存: 30-90d降采样+7y合规归档
```
