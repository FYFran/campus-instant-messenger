# 缉凶 — 合同框架版

## Red Lines（违反任一条 = 修复无效）

1. **不复现不修。** 不能稳定复现的 bug = 你找的"根因"是猜的。
2. **Counterfactual 必做。** "修了 X 后 bug 不再出现"——必须确认后才提交代码。
3. **不测试不修。** 修复前写出能复现 bug 的测试 → 确认 FAIL → 修复 → 确认 PASS。
4. **3 次修不好 → STOP。** 不是技术问题，是理解问题。重读证据，不要继续改代码。
5. **改代码前确认部署文件。** `deploy.py` 决定哪个文件跑在生产上。

## 合同链（每步产出 = 下一步门票）

```
[分类] → [证据] → [追踪] → [分析] → [修复] → [验证] → [记录]

规则：上一步的产出填写完整 → 下一步才能开始。
      任何一步的产出为空 → 后续全部无效。
      事后补填 → 标记 POST-HOC，置信度降一级。
```

| 步骤 | 产出（必须填写） | 未填写的后果 |
|------|----------------|-------------|
| 分类 | `Type: T___ 依据: ___` | 后续搜索无目标 |
| 证据 | `复现步骤: ___` + baseline 输出 | 追踪无起点 |
| 追踪 | `调用链: ___` + `Expected vs Actual 第一点: ___` | 分析无数据 |
| 分析 | `根因: ___ (conf: ___)` + Counterfactual 结果 | 修复无依据 |
| 修复 | 测试 FAIL→PASS 的完整输出 | 验证无基线 |
| 验证 | pre/post diff 对比 + `git diff --stat` | 记录不完整 |
| 记录 | `.fixes/{date}-{slug}.md` + bug-patterns 更新 | 经验丢失 |

## Bug 分类

| Type | 判定 | 特殊处理 |
|------|------|---------|
| T0 | 稳定复现 | 标准流程 |
| T1 | 加 print 消失/竞态/时序 | ⚠️观测者效应。print 仅 dev。prod 用只读观测 |
| T2 | ≥2 因素共触 | 每因素独立 CF。修一个不收工 |
| T3 | 无报错数据错 | 不变量+数据 diff |
| T4 | 昨天还好 | git bisect |
| T5 | 状态机异常 | 状态转移日志 |
| T6 | 特定环境 | 环境快照 diff（Docker/CI/k8s/裸金属） |
| T7 | 代码对需求错 | STOP。NOT_A_BUG。不修代码 |

## 工具回退

任何工具不可用 → 降级替代。无替代 → 标记 PARTIAL，继续。
- rg→Select-String(Win)/grep(Linux)
- codegraph→手动 Read 源码
- MCP 挂起→15s 超时→同语义工具切换→TIER1→BLOCK
- print() 不可用→日志/trace/时间线对比

## 模式

| 模式 | 触发 | 行为 |
|------|------|------|
| full | 默认 | 完整 7 步合同链 |
| quick | 快速/诊断 | 只分类+证据+分析，不修 |
| safe | 重要修复 | 修复后 [AWAIT APPROVAL]，人 approve 才继续 |
| emergency | 生产挂了 | 只红线 1-3，事后 24h 补链 |

## Emergency — 只 3 条

1. 复现 1 次（记录结果）
2. Counterfactual 确认（修后不再出现）
3. pre/post diff（0 新增回归）

事后 24h 内补全完整合同链。超时→标记 POST-HOC。
