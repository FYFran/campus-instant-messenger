# 轮回 v1.1 — Skill进化引擎

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 像训练神经网络一样优化Agent Skill文本。核心循环：评估→RED→诊断→有界编辑(≤4处)→执行验证→保留/回滚。
**Iron Law：** NO OPTIMIZATION ROUND WITHOUT BASELINE MEASUREMENT FIRST.
**红线：** 绝不修改CONSTITUTION段。绝不跳过人类确认改🔴tier skill。绝不LLM judge替代执行验证。绝不git reset --hard(只用git revert)。体积≤原始150%。循环≤3轮(不收敛→STOP升级人)。
**触发：** 优化skill / skill评分 / 改进skill / skill lab / skill实验 / skill质量
**边界：** Skill优化→轮回。Bug排查→缉凶。安全审计→铁壁。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas（来自真实失败）

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | 改了几行=更好了 | LLM judge准确率46.4% | 必须执行验证不靠LLM判分 |
| 2 | AI自动改完=完事 | AI生成skill +0pp，人类+16.2pp | 人类门禁不可跳过 |
| 3 | 改一个维度升了 | 另一个维度降了 | 全维度评分看总趋势 |
| 4 | 4处编辑都在改同一条规则 | 规则本身有问题 | 根因问题→重新设计非打补丁 |
| 5 | "99分不用改" | 某类型-15%被总分掩盖 | 分类型评分不只看总分 |
| 6 | 优化循环跑了好几轮 | 不收敛还在跑浪费钱 | 3轮不收敛→STOP升级人 |
| 7 | 没测基线就开始优化 | 不知道是否真的改进了 | RED Phase强制执行 |

## 理性化表（锻造时最常见的自我欺骗）

| 借口 | 现实 |
|------|------|
| "这个skill显然更好了" | 没跑分的"显然"=幻觉 |
| "改太小了不需要测基线" | 小改动=最大盲区 |
| "上次优化成功了这次也会" | 每次独立验证，不假设 |
| "时间不够先改了再说" | 违反Iron Law→删除重来 |
| "LLM judge说改进了" | LLM judge准确率46.4%≈抛硬币 |
| "就一处编辑不用full跑" | 一处编辑可降20%，必须full |
| "再跑一轮肯定收敛" | 3轮不收敛→问题在设计不在编辑 |

**违反Iron Law的字面=违反Iron Law的精神。不允许"技术性合规"。**

## GEPA风格进化循环（v1.1升级）

```
Phase 0: Registry Scan ← 新增
  扫描所有skill: 版本/最后优化日期/当前分数/使用频率
  优先级队列: (频率×失败率)排序，非最低分优先
  退役信号: 180天未用→标记archived

Phase 0.5: 评估 ← BugHuntBench full {skill}
  读执行轨迹，不只是看分数
  预算封套: 估算本轮总成本→>$0.50告知→>$5.00确认
  成本>skill预估价值→跳过

Phase 0.75: RED ← 新增(来自writing-skills精华)
  不加载目标skill，裸跑test-prompts
  记录agent在哪失败/为什么失败/失败模式
  建立"没有skill的基线"
  → 这是衡量skill真正价值的唯一方式

Phase 1: 诊断 ← 反思:"为什么失败?哪里失败+为什么"
  七维诊断卡: 触发准确/链完整/证据质量/修复正确/模型适配/体积健康/成本效率
  成本维度: skill执行token数是否在合理范围？静态分析+实测

Phase 2: 提案 ← 基于诊断提≤4处编辑
  编辑操作: 裁剪(最有效) > 替换 > 添加 > 重排
  每处编辑附: 依据(轨迹+失败)+预期效果+风险+如果失败的回退方案

Phase 3: 验证 ← BugHuntBench full {skill} AGAIN
  分维度对比前后分数
  盲审升级: 3 judge必须各找≥1个两版缺陷(防懒政)
  置信度<50%的judge→0.5票权
  确保3 judge不共享同一视角(防同质化)

Phase 4: 裁决
  提升→保留(你来确认)
  持平→保留(git ratchet不降)
  下降→自动git revert
  同finding 2轮不消→STOP升级人
  两validator冲突→调consensus-voting
```

**不加新指令。只做：裁剪冗余、替换弱约束、重排逻辑。**

## 七维诊断卡

| 维度 | 满分 | 判据 |
|------|------|------|
| 触发准确 | 1 | 该触发触发，不该触发不触发 |
| 链完整 | 1 | 每步产出非空，事前填非事后补 |
| 证据质量 | 2 | 复现可验证+CF有pre/post数据 |
| 修复正确 | 2 | 消根因不引入新问题 |
| 模型适配 | 1 | 目标模型上验证过 |
| 体积健康 | 1 | ≤原始150% |
| **成本效率** | **1** | **skill执行token≤基线+30%** |

## 反例黑名单

- 不修改CONSTITUTION

## 可成长性（元成长 — 优化器自己进化）

每次锻造完成后强制执行：
```
1. 锻造过程本身有什么可改进的？
2. 诊断卡有没有遗漏的失败维度？
3. 反例黑名单要不要加新条目？
4. 本次编辑模式(裁剪/替换/添加/重排)成功率？
5. 有没有"假收敛"——分数没降但实际变差了？
→ YES → .fixes/{date}-forge-meta.md
→ 你来确认→注入轮回自身
→ 积累"编辑模式库"——哪种编辑最有效
```

**技能成熟度追踪（L1-L5）：**
```
L1脚本 → L2参考 → L3宪政(有CONSTITUTION) → L4测试(有test-prompts+RED) → L5治理(有registry+版本+成本追踪)
每个被优化的skill标注当前等级。forge的使命是把skill从当前等级推向L5。
```
