# 缉凶 v3.0-alpha — 螺旋假设驱动

> 与 v2.5.1 的关键差异: 不预先分类。用"假设→验证→修正"循环替代"分类→追踪"线性链。
> T0-T7 是出口标签，不是入口关卡。

## CONSTITUTION

**核心功能：** 查代码+配置+skill 文本的 bug。螺旋循环：生成可证伪假设 → 快速验证 → 修正 → 逼近根因。
**安全约束：** 不复现不修。不 Counterfactual 不提交。不测试不修。3 轮假设未确认→质疑架构。
**触发：** bug / fix / error / crash / broken / 报错 / 不工作 / not working / bug report / 查skill
**边界：** 功能正确性→缉凶。性能→profiler。安全→铁壁。

---

## 螺旋合同链（替代线性链）

```
[Bug描述] → [H1: 初始假设] → [验证H1] → [推翻?] → [H2: 修正假设] → [验证H2] → [确认] → [深度追踪] → [修复] → [验证] → [记录]
    ↑                      ↑                        |
    |                      └── H1被推翻：记录推翻证据  |
    └────────────────────── H2也被推翻：第3轮，质疑架构 ──┘

规则:
1. H1 至少引用 bug 描述中 2 个具体词句作为依据
2. 每个假设必须可证伪: "如果我是对的，X 应该为真/假"
3. H1 被推翻→必须写"为什么H1错了"→从致命误判表(F1-F6)中找替代假设方向
4. 3 轮假设未确认→Red Line 4: 回到 bug 描述重读，可能根因类别完全错误
5. 假设确认后→进入深度追踪→根因→修复（与 v2.5.1 相同）
6. 根因确认后贴 T0-T7 标签（用于统计，不是入口分类）
```

---

## Red Lines

| # | 必须做的事 | 做完的证明 |
|---|-----------|----------|
| 0 | **假设可证伪** — H1 必须包含"如果对，___应该真。验证方法: ___" | 可执行的验证步骤 |
| 1 | **先复现再修** — 至少 1 次稳定复现 | 复现步骤 + baseline 输出 |
| 2 | **先 Counterfactual 再提交** | CF 含具体数字 |
| 3 | **先写能复现 bug 的测试再修** | FAIL→PASS 完整输出 |
| 4 | **3 轮假设未确认 → STOP** — 回到 bug 描述重读。可能根因类别完全错误 | 3 轮记录 + 推翻证据 |
| 5 | **修前确认文件映射** — 哪个文件跑在生产上 | 文件路径确认 |
| 6 | **找到第一个错误别停** — 继续往下挖 | 至少问一次 "还有没有更深的？" |

## 假设验证循环

### 第 1 轮：初始假设 H1

```
读 bug 描述 2 遍。问自己:
  "最可能出错的组件是什么？代码/配置/数据/网络？"
  "一个简单的、可立刻验证的假设是什么？"

H1 = ___
如果 H1 对，___ 应该为真/假。
验证方法: ___（最简单最快的验证方式）
验证结果: ___
H1 成立？ YES → 进入深度追踪 / NO → 记录推翻证据，进入 H2
```

**H1 信号→方向速查：**

| 信号 | 最可能方向（H1） |
|------|---------------|
| crash/panic/500 + 稳定复现 | 代码逻辑错误 (nil/边界/类型) |
| 间歇 + 同一操作 + 不同时间结果不同 | 时序/竞态/GC |
| 间歇 + 不同输入 + 结果不同 | 数据相关逻辑 (Contains/截断/空值) |
| 昨天能用 + 代码没改 + 重启过 | 配置变更 (nginx/env/端口) |
| 卡住/pending + 正常输入 | 状态机转移缺失 |
| 没报错 + 数据不对 + 每次一致 | 计算逻辑错误 |
| 本机 OK + CI/某环境挂 | 环境差异 (版本/OS/DB) |

### 第 2 轮：修正假设 H2

```
H1 被推翻。推翻证据: ___
从致命误判表(F1-F6)查找类似模式 → 哪个模式最像当前情况？
修正后的 H2 = ___
如果 H2 对，___ 应该为真/假。
验证方法: ___
验证结果: ___
H2 成立？ YES → 进入深度追踪 / NO → 进入 H3
```

### 第 3 轮：最后假设 H3

```
H1 和 H2 都被推翻。不继续假设——先质疑架构。
- 我是否误解了 bug 的根本性质？
- 这个 bug 是否跨越多个组件？
- 是否存在间接层（如 middleware/proxy/cache）我没考虑？

H3 = ___（基于重新理解）
验证结果: ___
H3 成立？ YES → 深度追踪 / NO → STOP。标记 PARTIAL，汇报已排除的方向。
```

## 深度追踪 + 修复（H 确认后，与 v2.5.1 同）

```
[H 确认] → [证据: 复现 + baseline] → [追踪: 调用链 + Expected vs Actual] → [分析: 根因 + conf + CF] → [修复: test FAIL→PASS] → [验证: diff + git diff --stat] → [记录: .fixes/ + bug-patterns]
```

## 出口标签 T0-T7（根因确认后贴）

| Label | 含义 | 典型根因 |
|-------|------|---------|
| **T0** | 稳定代码错误 | nil deref, off-by-one, type mismatch |
| **T1** | 时序/竞态 | missing await, TOCTOU, race condition |
| **T2** | 多因素/输入相关 | Contains too broad, multi-condition edge case |
| **T3** | 静默逻辑/计算错误 | truncation, wrong formula, N+1 query |
| **T4** | 配置/环境回归 | nginx port, env var lost, config revert |
| **T5** | 状态机缺陷 | missing state transition, stuck in loop |
| **T6** | 环境差异 | OS/version/DB difference |
| **T7** | NOT_A_BUG | behavior matches spec |

---

## T4 强制配置检查（H1 方向为"配置变更"时触发）

```
⚠ H1 怀疑配置问题 → 走这 3 步才能进入源码:
1. 配置 diff: nginx.conf / docker-compose*.yml / .env / systemd unit / 部署脚本
2. 端口/服务检查: 确认服务监听端口和 proxy_pass/upstream 目标匹配
3. 启动日志: 检查上次重启/部署日志，找 config loaded / reverted / warning
✅ 3 步全过且无异常 → 配置不是问题 → H1 被推翻 → 进入 H2(代码方向)
```

## 致命误判表（H 被推翻时的替代方向来源）

| # | 如果你之前认为... | 实际可能是... | 信号 |
|---|----------------|-------------|------|
| F1 | T3 静默数据错 | T1 时序/竞态 | 刷新又好了 |
| F2 | T3 静默数据错 | T2 输入相关 | 换参数值就正常 |
| F3 | T3 静默数据错 | T5 状态卡住 | 一直 pending |
| F4 | T1 竞态 | T3 逻辑错误间歇触发 | ~20次1次，非同一操作 |
| F5 | T0-T6 代码 bug | T7 NOT_A_BUG | 产品设计如此 |
| F6 | T3 代码 bug | T4 配置问题 | 昨天能用+重启后+代码没改 |

**用法：** H 被推翻时，查此表找最匹配的替代方向。不是预先查——是推翻后查。

| 模式 | 循环 | 触发 |
|------|------|------|
| **full** | 3 轮假设 + 深度追踪 + 修复 | 默认 |

| **safe** | full + [AWAIT APPROVAL] | 重要修复 |

## Gotchas

| # | 场景 | 动作 |
|---|------|------|
| 1 | 排查到根因 | 标 SUSPECT (conf<0.8) |
| 2 | T1 加 print bug 消失 | 诊断到此为止，print 不是修复 |
| 3 | 修了一个因素 | 继续修下一个，T2 所有因素独立 CF |
| 4 | 修完一个 bug | rg 全库搜同模式 |
| 5 | campus_check 过了 | 手动复现一次 |
| 6 | T3 静默数据错误 | CF 含数值对比（修前=X 修后=Y） |
| 7 | CF 写"修后 OK"无数字 | 回到证据步 |
| 8 | H 被推翻但没记录推翻证据 | 下一轮 H 无方向，浪费一轮 |

## Worked Examples（精简版 — 展示螺旋循环）

**示例 1: T4 配置回归**
```
Bug: API 全 401。昨天能用，重启过，代码没改。

H1: 配置丢失（方向: T4 配置变更）
  证伪: JWT_SECRET 未加载→api 全 401。验证方法: 检查 env + systemd unit。
  验证: Getenv("JWT_SECRET")=""。重启时间=首次 bug 时间。
  成立? YES → 标签 T4。根因: JWT_SECRET 未持久化。修复: systemd Environment=。

如果 H1 被推翻(env 有值):
  H2: token 生成逻辑被中间件拦截（方向: T3 代码逻辑）
  验证: curl 绕过中间件直接调 handler → token 正常
  成立? YES → 标签 T3。根因: middleware 错误修改了 header。
```

**示例 2: T3→T1 方向修正**
```
Bug: 积分偶尔不更新。刷新又好了。

H1: 计算逻辑错误（方向: T3）
  证伪: 如果计算错，应该每次错。但"刷新又好了"说明不是。
  验证: 连续 10 次→2 次不更新。H1 被推翻——不是计算错。
  推翻证据: "刷新又好了"+"不是每次都"=同一操作不同结果。

H2: 时序/竞态（方向: T1，查 F1: 刷新变化→T1）
  证伪: 如果竞态，同一操作不同时间结果应不同。
  验证: 追踪→coroutine 未 await→GC 随机回收。
  成立? YES → 标签 T1。根因: missing await。修复: 加 await at main.py:1095。
```

## 可成长性

| 触发 | 动作 |
|------|------|
| 同方向 H 被推翻模式出现 3+ 次 | → 追加到致命误判表 F7, F8... |
| 同假设验证方法 3+ 次无效 | → 更新"信号→方向速查" |
| 每个真实 bug 完成后 | → per_bug_results.tsv 追加 |
