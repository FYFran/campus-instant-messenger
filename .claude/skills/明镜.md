# Code Review v2.1 — 通用提交前审查

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 提交前代码审查。13类安全检查：Auth/RBAC/SQL注入/竞态/XSS/限流/输入验证/错误处理/数据暴露/审计日志/密码/依赖/跨后端一致。
**Iron Law：** NO APPROVAL WITHOUT READING THE ACTUAL FILES FIRST. 绝不凭记忆审。
**红线：** 绝不跳过审查直接批准。绝不凭记忆——先读文件。多后端必须都查。安全相关代码绝不自动修。一阶段不PASS不进二阶段。
**触发：** review / code review / 审查 / check my code / 检查代码 / 提交前
**边界：** 代码质量+安全→明镜。深度安全→铁壁。Bug→缉凶。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | "只改一行不用审" | 一行改错照样崩 | 一行也走完整审查 |
| 2 | Python后端过了Go没看 | 两个后端行为分歧 | 多后端必须都查 |
| 3 | semgrep没报=安全 | 规则覆盖不全 | 手动GREP补充 |
| 4 | happy path过了就批准 | 攻击者不走happy path | 每类check错误路径 |
| 5 | "上次审过了这次不用" | 新代码新漏洞 | 每次commit都审 |
| 6 | 服务端生成文件名报路径遍历 | 把安全编码当漏洞 | 只报告影响正确性/安全性的问题。占位符值≠缺陷。版本常量≠安全问题。每个finding问：用户会因此遇到bug/被攻击吗？ |
| 7 | 铁壁漏了=明镜也漏了 | 两个skill没知识共享 | 铁壁发现的SQL注入模式/密钥泄露模式→自动加入明镜审查清单。GEPA跨skill传播 |

## 3阶段门控（+Gate Functions——来自code-quality精华）

**规则：阶段1不PASS→不进阶段2。收敛：重扫直到连续两轮零新finding(≤3轮)。**

### 阶段1: 安全与正确性 🔴

每finding必须溯源到汇：
```
前提→追踪→验证→结论(SAFE/VULNERABLE/SUSPECT)
```
如果任何跳无法验证→CRITICAL。

Gate Functions（反模式门控——来自testing-anti-patterns精华）：
- 断言在mock对象上？→ "我在测真实行为还是mock存在？"→测mock→STOP
- 生产类有test-only方法？→ "这个只被测试用？"→移到test-utils
- Mock覆盖不全？→ "真的API返回什么字段？"→完整mirror
- 没写错误路径测试？→ "这个函数失败时会发生什么？"→补错误case

### 阶段2: 代码质量 🟠

13类检查+错误处理模式库：
- bare except:pass→绝不放过(最高频隐藏bug)
- 错误吞没：catch里只有log没有propagate→标记
- 错误泄露：响应里含内部路径/栈追踪→CRITICAL
- 无错误处理：外部调用(DB/API/文件)无try-catch→标记

### 阶段3: 领域完整性

跨后端一致+依赖CVE+收敛验证

## 可成长性

每次审查完成后：
```
1. 13类之外的新漏洞模式？→ 提案新类别
2. 有什么"常识"agent不知道？→ Gotchas候选
3. 假阳性有没有？→ 精度校准
4. Gate Function有没有拦住真的问题？→ 验证有效性
→ 轮回采集→确认→注入
```
