# 试金石 v1.1 — 测试锻造

## CONSTITUTION（不可被 forge 编辑）

**核心功能：** 完整测试栈：TDD纪律→反模式门控→集成测试→边界诚实。测试=规格，不是验证。
**Iron Law：** NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST. 不先写测试=删除代码重来。
**红线：** 绝不测mock行为。绝不往生产类加test-only方法。绝不在不理解依赖时mock。测试必须看它FAIL再修。一次只测一个行为(test名有"and"→拆分)。
**触发：** 写测试 / test / 测试 / TDD / add test / coverage / 测试覆盖 / write tests
**边界：** 测试→试金石。Bug修复→缉凶(缉凶Red Line3调试金石)。安全→铁壁。
**模型：** deepseek-v4-pro。换模型→重跑BugHuntBench quick。

## Gotchas（来自4个测试来源skill精华）

| # | 症状 | 根因 | 教训 |
|---|------|------|------|
| 1 | 测试一写就PASS | 没看FAIL就写实现 | RED强制:先看FAIL确认原因对 |
| 2 | mock占了测试>50% | 设计太耦合 | 重设计而非加mock |
| 3 | 测试名有"and" | 测了多个行为 | 拆成多个单一行为测试 |
| 4 | "太难测了" | 接口设计有味道 | 简化接口不硬写复杂测试 |
| 5 | E2E测试测业务逻辑 | 测试放错层级 | 单元测逻辑集成测边界E2E测流 |
| 6 | 没写错误路径测试 | 只测了happy path | 每个函数问"失败会发生什么" |
| 7 | 集成测试放错层 | 单元能测的放到集成 | 往下能测的不往上推。单元测逻辑→集成测边界→E2E测流 |

## 六条铁律

| # | 铁律 | Gate Function |
|---|------|---------------|
| 1 | 无FAIL测试不写生产代码 | "测试第一次跑就PASS？"→没测到bug→回RED |
| 2 | 测行为不测实现 | "改了实现测试还PASS吗？"→不PASS？→测试绑定了实现 |
| 3 | Mock什么不拥有什么 | "这是我们的代码还是外部边界？"→我们的→不mock |
| 4 | 每个测试必须先看它FAIL | "失败原因对吗？"→不是预期的失败→重写测试 |
| 5 | 测试痛苦=设计痛苦 | "测试setup>50%文件？"→设计有问题→先重构 |
| 6 | 知道你不测什么 | "这个测试不覆盖什么？"→写下来→边界诚实 |

## 四支柱

### 支柱1: RED-GREEN-REFACTOR（来自TDD）

```
RED:   写FAIL测试→确认失败原因正确
GREEN: 最少代码让测试PASS(允许hardcode/跳过边界)
REFACTOR: 清理重复→改善设计→测试必须保持PASS
         任何REFACTOR期间FAIL→立即回退→小步重来

验证清单(8项，全通过=完成):
[ ] 每个函数/方法有测试
[ ] 看了每个测试先FAIL
[ ] 最少代码让测试PASS
[ ] 所有测试PASS
[ ] 输出干净(无警告/跳过)
[ ] 使用真实代码(mock仅外部边界)
[ ] 边界case覆盖
[ ] 测试名无"and"
```

### 支柱2: 金字塔放置（来自integration-testing）

```
单元测试:   纯逻辑、算法、验证 → 最快最多
集成测试:   数据库/API/文件系统边界 → 真实组件+可控输入
E2E测试:    关键用户流程 → 最少最慢

规则: 往下能测的不往上推。E2E测业务逻辑=浪费。
覆盖率: 关键路径≥80%
目录: tests/unit/ + tests/integration/ + shared conftest.py
```

### 支柱3: Gate Functions（来自testing-anti-patterns）

```
Mock前:     "真的方法有什么副作用？"→不知道→不mock
            Mock响应是否包含真实API的所有字段？→不全→补全
生产类:    "这个方法只被测试调用？"→是→移到test-utils
Mock后:    去掉mock→测试还测真实行为吗？→不测→写集成测试
```

### 支柱4: 边界诚实（来自stress-test）

```
每个测试文件声明: "本测试不覆盖___"
至少2个竞争机制: 如果无法想出一个反例→理论太宽
尺度扰动检查: 单元PASS但集成FAIL？→为什么？
```

## 测试质量严重度分级

| 级别 | 定义 | 示例 |
|------|------|------|
| 🔴 CRITICAL | 测试给出虚假安全感 | 恒真测试、mock覆盖被测逻辑、断言总返回true |
| 🟠 HIGH | 测试覆盖率盲区 | 关键路径无测试、错误路径零覆盖、竞态条件未测 |
| 🟡 MEDIUM | 测试设计问题 | 测试名有"and"、mock过度、test-only方法污染生产代码 |
| 🟢 LOW | 风格/可读性 | 测试命名不规范、setup重复可提取 |
