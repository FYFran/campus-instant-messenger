## 二、外部智慧精华

### 2.1 Anthropic 核心教训

**教训1：评分结果，不评过程。**
```
❌ 检查 Agent 是否调了正确的工具、按正确顺序
✅ 检查最终代码是否正确、测试是否通过
```

**教训2：pass@k ≠ pass^k。**
- 75%单次成功率 → 仅42%概率连续3次成功
- 缉凶的6.1/8意味着：如果连跑3个bug，至少1个会出问题的概率是 `1 - 0.76³ = 56%`

**教训3：基础设施噪音可造成 6pp 分数差。**
- 需要标准化运行环境、硬杀阈值、资源校准

**教训4：子Agent调度评估。**
```
DispatchCorrectness — 选了正确的Agent类型？
ScopeFidelity — Agent有没有越权调工具？
ResultIntegration — 主Agent有没有读子Agent的返回？
```

### 2.2 DSPy 核心教训

**教训1：程序化 > 手工。**
- 手工调 prompt：每次改一两句，跑一遍，看效果。慢、不稳定。
- DSPy：把 prompt 当代码编译，自动搜索最优 instruction + few-shot example 组合。

**教训2：优化器选型取决于数据量。**
| 数据量 | 优化器 | 适用场景 |
|--------|--------|---------|
| < 20 样本 | BootstrapFewShot | 新 skill，数据少 |
| ~50 样本 | BootstrapFewShot + Random Search | 中等积累 |
| 200+ 样本 | MIPROv2 | 成熟 skill，数据多 |

**教训3：小模型收益大，大模型收益小。**
- Qwen3 (local): DSPy 提升 +12.5%
- GPT-4o (frontier): DSPy 提升 0%
- 我们的 deepseek-v4-pro 是 frontier 级 → DSPy 直接优化 prompt 文本可能收益有限
- **但** DSPy 的 few-shot example 选择优化对 frontier 模型依然有效

**教训4：训练数据质量是最关键组件。**
- "80% of emails matched human quality" 的前提是有高质量的人工反馈数据

### 2.3 SWE-bench 方法论精华

**教训1：公共基准分数不迁移到你的代码库。**
- "SWE-bench Verified is the starting line, not the finish"
- 需要在**自己的代码库**上建立评估

**教训2：五维评估。**
| 维度 | 含义 | 我们对应 |
|------|------|---------|
| Golden-PR replay | diff 是否通过 review？ | ❌ |
| Tool-call correctness | 工具调用顺序对吗？ | ❌ |
| Multi-file coherence | 所有调用点都改了吗？ | ❌ |
| Plan coherence | 计划是否符合意图？ | ❌ |
| Rollback discipline | 失败后回滚还是堆叠？ | ❌ |

**教训3：Benchmark Mutation。**
- GitHub issue → 用户自然语言描述的转换会显著改变评估结果
- 公共基准高估 agent 能力 >50%

**教训4：200K+ 次运行的基础设施教训。**
- 吞吐墙、状态碰撞（隔离）、可靠性税（可恢复性）
- 需要解耦的生成/评估管线

### 2.4 2026 最新：Holistic Agent Evaluation (arXiv 2605.14865)

> **"Evaluation methodology, not model capability, is the bottleneck"**
> 同一 frontier 模型在框架内评分比单体 judge 高数倍

关键：**自顶向下 agent 级诊断 + 自底向上 span 级评估** 的组合。

## 三、瓶颈分析

### 瓶颈1：L2 盲区 — 8/11 skill 无行为基线

**症状：** 只有铁壁、明镜、缉凶有 L2 基线。其他8个skill改了之后不知道变好变坏。

**根因：** L2 跑一次 full benchmark 成本高（~50K token/bug），没有自动化触发机制。

**影响：** 门神、破阵、布阵、火眼等 skill 的修改都是"凭感觉"，飞轮对它们形同虚设。

### 瓶颈2：缉凶 6.1/8 — 最常用 skill 基线最低

**症状：** 40个bug，平均漏 ~2分/8分。按 Anthropic 标准，这意味着debug agent有显著盲区。

**需要搞清楚的问题：**
- 6.1 分主要丢在哪几个维度？
- 是某个 T-Type 特别差还是全面偏低？
- 是证据链不完整还是根因定位不准？
- agent 的 transcript 里实际发生了什么？

### 瓶颈3：只有分数，没有 transcript

**症状：** 知道缉凶得分6.1，但不知道**为什么**只得6.1。

**Anthropic 第6步明确说：不许只看分数不看卷子。**

### 瓶颈4：飞轮半闭环 — 只能检测不能行动

**症状：** L0+L1 能检测漂移，生成 GEPA 提案。但 GEPA → 轮回 → 改 skill → 验证 这条链没有自动化。

### 瓶颈5：缺少生产反馈

**症状：** Anthropic 第9步 — eval 分数必须与生产表现校准。我们没有任何生产数据。

## 四、三阶段方案

### Phase 1: 全量基线 + transcript 审计（本周）

**目标：消灭 L2 盲区，建立行为基线，读第一次 transcript**

#### 1.1 跑全量 L2 baseline（8个缺失 skill）

```
skill           bugs   估计token    估计时间
门神 v2.1       3      ~150K        ~3min
破阵 v2.1       3      ~150K        ~3min
布阵 v3.0       3      ~150K        ~3min
火眼 v1.1       3      ~150K        ~3min
试金石 v1.0     8      ~400K        ~8min
天眼 v1.0      (无bugset，跳过)
架构师 v1.0    (无bugset，跳过)
轮回 v1.1      (无bugset，跳过)
------------------------------------------------
合计            20     ~1M tokens   ~20min
```

**实现：** `python .claude/benchmarks/bughunt/bughunt_ci.py --skill 门神 --mode full`
循环跑每个缺失skill，结果写入 `baselines.json`。

#### 1.2 缉凶 transcript 深度审计

**这是 Anthropic 第6步 —— 整个方案最关键的一步。**

方法：
1. 从 40 个缉凶 bug 中，挑分数最低的 10 个
2. 对每个低分 bug，拉完整 agent transcript（tool calls + reasoning）
3. 用 GEPA 框架分类失败模式：

```
G EPA 审计矩阵：
├── G (Generate pattern): 失败属于哪种模式？
│   ├── 分类错误：T-Type 判错
│   ├── 链断裂：7步缺步
│   ├── 证据弱：file:line 缺失
│   ├── 根因偏：方向对但定位错
│   ├── 反事实缺失：没有pre/post对比
│   └── 修复失败：改了但测试不过
├── E (Evaluate impact): 每种模式影响多少分？
├── P (Propose fix): 针对最高影响模式，怎么改 contract？
└── A (Act): 小步改 → 重跑 → 验证
```

**输出：** `缉凶_v3.1_transcript_audit.md` — 失败模式分布 + 根因统计

#### 1.3 建立结构基线（为 L2 触发打基础）

```
structure_baselines.json 已覆盖:
├── L0: file exists, frontmatter, size, sections ✅
├── L1: gotcha count ratchet, constitution, red lines ✅
└── L2: (新增) score baseline + drift threshold
```

**新增：** 在 `drift_check.py` 支持 `--mode l2` 对比当前跑分 vs 基线。

### Phase 2: 缉凶 v3.1 → v3.2（下周）

**目标：基于 transcript 审计，针对性优化缉凶 contract，6.1 → 7.5+**

#### 2.1 DSPy 风格程序化优化

基于 Phase 1 的 transcript 审计结果，选择性应用：

**策略A：Gotcha 表扩展（确定性收益）**
- 从失败 transcript 中提取新的失败模式
- 添加到 缉凶.md 的 Gotchas 表
- 这已经被 轮回 的优化历史验证有效（铁壁 63→99, 火眼 43→100）

**策略B：Few-shot example 注入（DSPy BootstrapFewShot 等价）**
- 从高分 transcript 中提取最佳实践的 trace
- 注入到 contract 的 example 部分
- 给 agent 提供"正确答案长什么样"的参考

**策略C：分类决策流优化（DSPy MIPRO 等价）**
- 如果 transcript 显示 T-Type 分类经常错
- 重写分类决策树，增加区分度更高的判别条件
- MIPROv2 等价：同时优化 instruction + example 组合

**策略D：红线/Iron Law 收紧**
- 如果 transcript 显示某类错误反复出现
- 添加新的红线约束
- 例如："NO ROOT CAUSE WITHOUT FILE:LINE + LINE NUMBER"

#### 2.2 验证方法：A/B 对照

```python
# 修改前基线
bughunt_ci.py --skill 缉凶 --mode full  # 记录 baseline = 6.1

# 改 contract
Edit 缉凶.md  # 应用 transcript 审计驱动的改进

# 修改后验证
bughunt_ci.py --skill 缉凶 --mode full  # 新分数

# 统计检验
if new_score - baseline >= 0.5:  # 最小实际改善阈值
    update baselines.json
else:
    rollback + 重新分析
```

#### 2.3 pass@k 测量

当前只用单次分数。增加 pass@k 指标：

```
pass@1 = 单次成功率
pass@3 = 3次中至少一次全对
pass^k = k次连续全对 (Anthropic 用于面向用户的agent)

缉凶目标:
  pass@1: 76% → 85%+
  pass@3: 待测
```

### Phase 3: 飞轮 L2 自动化 + L3 采样 + 生产校准（两周）

**目标：闭环自动化，飞轮真正转起来**

#### 3.1 L2 自动触发

```yaml
触发条件:
  - skill .md 文件变更 → git diff 检测
  - 自动跑 quick mode (0 token, <1s, L0+L1)
  - 如果 L0+L1 通过 → 自动跑 full mode
  - 分数 vs baseline 对比
    - 上升 >5% → 建议更新 baseline
    - 下降 >5% → 阻止合并 + 触发 GEPA

实现位置:
  - .claude/hooks/ → PostToolUse hook
  - 检测 Edit 目标是否在 .claude/skills/ 下
  - 是 → 跑 drift_check.py --skill X
```

#### 3.2 L3 对抗采样

```
策略: 不跑全量 L3（太贵），采样抽查
  - 每周随机抽 3 个 bug
  - 用 Sonnet 评审 DeepSeek 的输出
  - 标记: REAL / REAL* / TEMPLATE / WRONG
  - 如果 WRONG 率 > 20% → 扩大抽查到全量
```

#### 3.3 DAG 链执行

DAG 已在 `references/skill_dag.json` 中定义。实现链式执行：

```
例: "改 campus_go 的 activities.go"
  → 明镜(review diff) → 试金石(run tests) → 铁壁(audit) 
  → 门神(gate) → 布阵(deploy)
  
每个环节产出传给下一环节，任一失败则停止。
```

#### 3.4 生产校准（最小可行版）

```
Anthropic 第9步: evals + production monitoring + user feedback

我们能做到:
  1. 记录每次 agent 调用的实际结果（通过/失败/部分）
  2. 每月对比 eval 分数 vs 实际成功率
  3. 偏差 >10% → eval 需要更新（可能过拟合）
```

#### 3.5 破阵 pass@2

破阵目前 pass@1。增加 pass@2 需要：
- 每个 bug 跑 2 次
- 统计至少 1 次成功的概率
- 与 baseline 对比

**预留任务，Phase 3 执行。**

## 八、理论根基总结

本方案直接借鉴的外部智慧：

1. **Anthropic 9-Step Roadmap** (2026): 步骤6（读transcript）和步骤7（监控eval饱和）是我们最大的差距，也是 Phase 1-2 的核心
2. **DSPy Programmatic Optimization** (Omar Khattab, Stanford): 程序化 > 手工，BootstrapFewShot + MIPROv2 选型策略，few-shot example 对 frontier model 仍有效
3. **Holistic Agent Evaluation** (arXiv 2605.14865, 2026): "评估方法论而非模型能力才是瓶颈"，自顶向下诊断 + 自底向上 span 评估
4. **Claw-SWE-Bench** (arXiv 2606.12344, 2026): Adapter 设计对分数影响巨大（19% → 73%），标准化运行条件至关重要
5. **Saving SWE-Bench** (CAIN 2026): 公共基准高估能力 >50%，自建代码库评估才是真
6. **Scaling Agentic Evaluation** (AI21, 2026): 200K+ 运行的基础设施教训——隔离、可恢复性、解耦管线
7. **Anthropic's Own Regressions** (2026): 即使最先进的评估车间也会漏回归——缩短用户信号到评估覆盖的延迟
